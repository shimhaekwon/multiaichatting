## 1. 개요

### 1.1 목적
사용자가 둘 이상의 LLM과 동시에 대화하며 브레인스토밍하여, 머릿속에 정리되지 않은 아이템 또는 프로젝트를 구조화하는 것을 지원한다.

### 1.2 배경
단일 LLM 대화는 관점이 한정적이다. 여러 LLM이 서로의 의견을 참조하며 토론하면 다각적 시각으로 아이디어를 발전시킬 수 있다.

### 1.3 범위
- 포함: 크롬 확장 기반 멀티 LLM 라운드형 대화 오케스트레이션, 라운드 제한, 비밀 채팅
- 제외: 공식 API 연동(본 시스템은 브라우저 로그인 세션 활용 방식), LLM 응답 품질 보정

### 1.4 용어 정의
| 용어 | 정의 |
|------|------|
| 라운드(Round) | 사용자 request 1회 + 참여 LLM 각각의 response 1회로 구성되는 대화 단위 |
| 오케스트레이터 | 라운드 진행, 응답 수집, 타 LLM 전달을 제어하는 핵심 모듈 |
| 비밀 채팅 | 전체 브레인스토밍 맥락과 격리되어 특정 LLM과 사용자만 진행하는 별도 대화 |

## 2. 시스템 개요

### 2.1 플랫폼
- 크롬 확장 프로그램 (Manifest V3)
- chathub와 유사하게, 크롬 브라우저에 로그인된 Google 계정으로 접근 가능한 LLM 웹 서비스와 대화

### 2.2 주요 구성
- Background Service Worker: 라운드 상태머신 / 오케스트레이션
- Content Scripts: LLM 사이트별 어댑터 (프롬프트 주입, 응답 추출)
- UI (Side Panel/Popup): LLM 선택, 라운드 설정, 대화 표시, 비밀 채팅

## 3. 기능 요구사항

| ID | 구분 | 요구사항 | 우선순위 |
|------|------|----------|:------:|
| FR-01 | 참여 | 둘 이상의 LLM이 브레인스토밍에 참여할 수 있다. | 필수 |
| FR-02 | 라운드 규칙 | 한 LLM은 한 라운드당 1회만 response 한다. (라운드 내 1회 제한) | 필수 |
| FR-03 | 응답 길이 | LLM 응답은 1회 1000 토큰을 초과하지 않는다. 라운드마다 request에 1000토큰 제한 지시문을 자동 추가한다. | 필수 |
| FR-04 | 응답 순서 | 라운드 내 LLM들의 응답 순서는 제한하지 않는다. | 필수 |
| FR-05 | 응답 전달 | 한 라운드 종료 후, 사용자의 다음 request와 함께 각 LLM에게 "자신을 제외한" 타 LLM 응답을 라운드 단위로 전달한다. | 필수 |
| FR-06 | 라운드 제한 | 시작 시 진행 횟수를 3~9 범위로 지정한다. | 필수 |
| FR-07 | 중단 | 진행 중 언제든 사용자가 stop 할 수 있다. | 필수 |
| FR-08 | LLM 접근 | 크롬에 로그인된 Google 계정으로 접근 가능한 LLM과 대화한다. (chathub 방식) | 필수 |
| FR-09 | LLM 목록 | 선택 가능한 LLM 목록은 top 8 까지만 표시한다. | 필수 |
| FR-10 | 비밀 채팅 | 전체 대화와 무관하게, 특정 LLM과 사용자만 별도로 대화하는 기능을 제공한다. (전체 맥락과 완전 격리) | 선택 |

### 3.1 응답 전달 형식 (FR-05 상세)
각 LLM에는 자신의 응답을 제외한 나머지 LLM 응답을 JSON 라벨 형태로 전달한다.

```
gemini  ← {"chatgpt":"그건 아닌거 같은데", "claude":"그럴 수도 있어"}
claude  ← {"chatgpt":"그건 아닌거 같은데", "gemini":"그럴 수도 있어"}
chatgpt ← {"gemini":"고민스럽네", "claude":"그럴 수도 있어"}
```

### 3.2 라운드 진행 흐름
1. 사용자가 참여 LLM 선택 (최소 2개) 및 라운드 횟수(3~9) 지정
2. 사용자 request 입력
3. 각 LLM에 [1000토큰 지시문 + request + 타 LLM 직전 라운드 응답(자신 제외)] 전달
4. 각 LLM 응답 수집 (라운드당 1회, 순서 무관)
5. 라운드 카운트 증가 → 다음 라운드 반복
6. 지정 횟수 도달 또는 사용자 stop 시 종료

## 4. 비기능 요구사항

| ID | 구분 | 요구사항 |
|------|------|----------|
| NFR-01 | 호환성 | Manifest V3 기반, 크롬 최신 안정 버전 지원 |
| NFR-02 | 유지보수성 | LLM 사이트별 어댑터를 설정/모듈로 분리하여 UI 변경 대응 용이하게 한다 |
| NFR-03 | 안정성 | 응답 완료(스트리밍 종료) 감지 로직을 확보한다 (전송 버튼 복귀 / DOM mutation 정지 등) |
| NFR-04 | 격리성 | 비밀 채팅 데이터는 메인 브레인스토밍 컨텍스트에 혼입되지 않는다 |
| NFR-05 | 사용성 | 진행 상황(현재 라운드/총 라운드), stop 버튼을 항상 노출한다 |
