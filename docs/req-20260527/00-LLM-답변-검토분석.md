# 5개 LLM 답변 검토/분석

> 기준: user-request.md v2.0 요구사항 대비 각 LLM의 10-기본 / 20-핵심 / 30-종합 답변 충실도 및 차이점 분석

---

## 1. Claude

### 10-기본
- **요구사항 대비 충실도**: ★★★★★
- 매우 간결하면서도 핵심 7개 항목(로그인 세션, 3LLM 탐색, 2개 선택 필수, 3/9라운드, 동기화, 타임아웃, 저장)을 빠짐없이 포함
- ChatHub 개선 베이스 명시
- 출력 형식 강제(구조화, 1500토큰) 포함
- **no guess**: 추가 추정 없음. 요구사항 범위 내에서만 작성

### 20-핵심
- **요구사항 대비 충실도**: ★★★★★
- 모듈 아키텍처 베이스 → LLM Adapter Layer → 세션 관리 표 → 라운드 엔진 정의 순으로 체계적
- `Promise.all` 패턴 언급, Request Payload Spec 항목화
- IndexedDB 스키마를 4개 Object Store(sessions, rounds, responses, userInputs)로 구체화
- **미결 항목 3건 명시**: 1500토큰 제한 enforcement, DOM 변경 대응, idempotency — 이는 no guess 원칙에 충실한 태도
- 저장 스키마에 tokens, latency 등 부가 필드 포함 → 실제 구현에 유용

### 30-종합
- **요구사항 대비 충실도**: ★★★★★
- 비전문가 대상임을 명시, "왜"를 포함한 설명
- 플로우 다이어그램(텍스트 기반)으로 R1→R2→R3 시각화
- "몰래 채팅 MVP 이후" 명시
- **구현 순서 10단계 제안** — 1단계 ChatHub 분석부터 10단계 몰래 채팅까지 단계화
- 1500토큰 제한에 ⚠️ 표기로 미확정임을 강조
- 제약사항을 ❌ 아이콘으로 가시화

### 종합 평가
요구사항에 가장 충실하게 매핑됨. 4무 원칙(빠짐/틀림/다름/더함)을 가장 잘 지킴. 특히 20-핵심의 DB 스키마와 미결 항목 명시가 강점.

## 2. ChatGPT

### 10-기본
- **요구사항 대비 충실도**: ★★★★☆
- 7개 섹션(핵심개념, 다중LLM, 라운드, 사용자제어, 저장, 기술조건, 개발원칙)으로 구조화
- "API 없는 LLM 협업"이라는 표현으로 차별화
- 전체 선택/일부 선택/세션 유지 등 구체 항목 포함
- **누락**: 타임아웃(5번 항목) 언급 없음. ChatHub 베이스 언급 없음

### 20-핵심
- **요구사항 대비 충실도**: ★★★★☆
- 포함/제외 범위를 명확히 구분 (scope in/out)
- R1/R2/R3+ 입력과 목적을 각각 분리 정의
- LLM 전달 정보에 "현재 회차, 최대 회차, timeout 정보 포함 필요" 명시
- **누락**: IndexedDB 구체 스키마 없음. LLM Adapter Layer 개념 없음. 미결 항목 없음
- 시스템 목적/범위를 먼저 정의하는 top-down 접근

### 30-종합
- **요구사항 대비 충실도**: ★★★★☆
- 목표 기능을 사용자 중심으로 기술 ("사용자가 할 수 있어야 하는 것")
- R1/R2/R3의 목적을 각각 "독립 사고 확보", "상호 검토", "수렴 및 최종 정리"로 명확히 구분
- 응답 구조화 예시(제목/요약/핵심내용/결론)까지 제시
- **누락**: 몰래 채팅 MVP 이후 언급 없음. 구현 순서 없음

### 종합 평가
구조화와 문서 체계는 가장 깔끔함. scope in/out 구분이 명확하고 R1~R3 목적 정의가 구체적. 그러나 구체적 아키텍처 요소(Adapter, DB 스키마)와 미결 항목 식별이 부족.

## 3. DeepSeek

### 10-기본
- **요구사항 대비 충실도**: ★★★☆☆
- 4개 항목으로 매우 간결 (타 LLM 대비 가장 짧음)
- 핵심 4가지만 압축: 연동, 라운드, 선택, 저장
- **누락**: 출력 형식 강제, 타임아웃, 개발 원칙, ChatHub 베이스, 기술 제약 다수 누락
- "오프라인"이라는 표현 사용 — "로컬 전용"이 정확함 (오프라인은 네트워크 미연결 의미)

### 20-핵심
- **요구사항 대비 충실도**: ★★★☆☆
- 유일하게 실제 코드 스니펫 포함 (RoundController 클래스, IndexedDB JSON 스키마)
- `chrome.cookies` API, `Promise.allSettled()`, `setTimeout()` 등 구체 기술 언급
- **누락**: LLM Adapter, 사용자 개입 정책, 응답 강제 규칙, 저장 스키마 상세, 미결 항목
- 코드 조각은 있지만 설명과 구조화가 부족하여 "전문가 구현 가능" 수준에 미달

### 30-종합
- **요구사항 대비 충실도**: ★★☆☆☆
- mermaid 다이어그램 시도는 좋으나 매우 간략
- try-catch 스타일의 에러 핸들링 코드 포함
- **전체 길이가 매우 짧아** 30-종합 수준에 크게 미달. 사용자 시나리오 설명, 제약사항, 저장 방식 등 대부분 항목 누락

### 종합 평가
가장 부실함. 코드 스니펫을 포함한 점은 독특하나, 전반적인 항목 누락이 심각. 10-기본은 그나마 OK지만 20-핵심은 전문가용이라기엔 내용이 빈약하고 30-종합은 사실상 요구사항 절반이 누락됨.

## 4. Gemini

### 10-기본
- **요구사항 대비 충실도**: ★★★★☆
- 요구사항 모든 항목을 하나의 문단형 설명으로 커버 (다른 LLM과 달리 글머리 위주가 아님)
- R1/R2/R3 설명이 명확하고 구체적
- ChatHub 오픈소스 기반 명시
- **문제점**: "개요 수준 1장" 요청에 비해 너무 장황함. bullet point 대신 긴 문단으로 가독성 낮음

### 20-핵심
- **요구사항 대비 충실도**: ★★★★★
- 6개 모듈(Session Manager, Round Engine, User Interaction, Timeout, Storage, Response Formatting)로 완전 분해
- 각 모듈별로 상세 워크플로우와 예외 처리까지 포함
- "MVP 이후" 항목을 별도 명시하여 스코프 관리 명확
- Prompt 메타데이터 주입 구체화 (`[제한시간] + [현재차수/제한차수]`)
- **누락**: IndexedDB 구체 스키마 없음. 미결 항목 없음

### 30-종합
- **요구사항 대비 충실도**: ★★★★★
- 사용자 시나리오를 1~8단계로 완전 분해, 단계별 동작 방식 상세 기술
- "연결 탐색 실패 시 대응 워크플로우"를 2중 조건까지 구체화
- 라운드 제어를 "3단계: 라운드 규칙"으로 설명 + 조기 종료/연장 포함
- "주의(구현 금지): 몰래 채팅 MVP 이후" 명시
- 토큰 제한을 "글자수 제한"으로 비전문가 용어 사용
- **누락**: 구현 순서 제안 없음

### 종합 평가
가장 상세하고 정성적인 답변. 20-핵심의 모듈 분해와 30-종합의 시나리오화가 탁월. 단점은 다소 장황하고 개요 수준 문서에도 문장형 서술이 많아 요구사항 포맷과 거리가 있음. 미결 항목 식별이 없는 점은 아쉬움.

## 5. Mistral

### 10-기본
- **요구사항 대비 충실도**: ★★★★☆
- "ChatHub 크롬 확장 플러그인으로 실행" 명시 (다른 LLM 대비 베이스를 강조)
- R1/R2/R3 각각 설명
- **누락**: 출력 형식 강제, 타임아웃 구체 수치(5분)

### 20-핵심
- **요구사항 대비 충실도**: ★★★☆☆
- Mistral만의 독특한 해석 존재:
  - "타 LLM 재질의: R3 이후에만 가능" — 원본 요구사항에는 없는 제약 (no guess 위반 가능성)
  - "응답 형식: JSON/XML 기반" — 원본에는 "구조화"만 명시, JSON/XML은 추정
- 크롬 버전 "Chrome 100+" 명시 (요구사항에는 "최신 안정 버전")
- **누락**: 라운드 엔진 상세, IndexedDB 스키마, LLM Adapter, 미결 항목
- 9번 개발 원칙 항목으로 점프 (번호 불일치)

### 30-종합
- **요구사항 대비 충실도": ★★★☆☆
- "도메인별 LLM 우선순위" 언급 — 원본 요구사항에 없는 내용 (추정)
- 재질의 제약을 "R3 이후에만 타LLM 재질의"로 원본과 다르게 해석
- ChatHub 플러그인 UI 참조 URL 포함 — 그러나 URL은 예시
- **누락**: 시스템 개요 후 상세 시나리오가 부족. 라운드 진행 흐름 설명이 빈약

### 종합 평가
전반적으로 요구사항 원본에 없는 내용이 포함됨 (no guess 위반 소지). Chrome 100+, JSON/XML, 도메인별 우선순위 등은 요구사항 명세에 없는 추정 요소. 20-핵심은 타 LLM 대비 깊이가 부족하고 30-종합은 누락과 추정이 혼재.

## 8. 총평

### 가장 충실한 답변: Claude
요구사항 원본을 가장 정확히 반영했으며, no guess 원칙을 철저히 지킴. DB 스키마와 미결 항목 식별이 타 LLM 대비 돋보임.

### 가장 상세한 답변: Gemini
20-핵심의 모듈 분해와 30-종합의 시나리오 설명이 가장 풍부함. 다만 개요 문서가 장황하고 미결 항목 식별이 없음.

### 가장 부실한 답변: DeepSeek
10-기본과 20-핵심에 코드 스니펫이 포함된 점은 독특하나 전반적 항목 누락이 심각. 특히 30-종합은 요구사항 절반 이상이 빠짐.

### no guess 위반 소지: Mistral
"R3 이후에만 타LLM 재질의 가능", "JSON/XML 기반", "도메인별 LLM 우선순위" 등 원본에 없는 내용 다수 포함.

### 구조화 최우수: ChatGPT
Scope in/out 구분, R1~R3 목적 정의, 문서 계층 구조가 가장 체계적. 다만 기술 깊이가 부족하고 ChatHub 베이스와 미결 항목 누락.
