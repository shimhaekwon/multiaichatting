## 📄 파일 2: `20-핵심-고객요구.md`

```markdown
# 멀티-LLM 라운드 토론 시스템 — 핵심 요구사항 (전문가용)

## 1. 시스템 아키텍처

### 1.1 베이스
- **Base**: ChatHub (E:\workspace\chathub) fork & enhancement
- **Type**: Chrome Extension (Manifest V3)
- **Storage**: IndexedDB (local-only, no server)
- **Auth**: Browser session cookie 기반, API key 미사용

### 1.2 LLM Adapter Layer
- Claude / ChatGPT / Gemini 각각 web session adapter 구현
- 로그인 상태 detection: content script로 각 사이트 cookie/session 확인
- 미로그인 시: 해당 LLM 로그인 페이지 안내 → 유효 LLM 0개 시 default LLM 계정 생성 가이드

## 2. 세션 & 선택 관리

| 항목 | 사양 |
|---|---|
| 최소 선택 LLM | 2개 (validation 강제) |
| 선택 상태 persistence | chrome.storage.local |
| 라운드 도중 변경 | 차단, 라운드 경계에서만 mutation 허용 |

## 3. 라운드 엔진

### 3.1 라운드 정의
```
R1 (Blind):       사용자 프롬프트만 전달, 타 LLM 응답 미공유
R2 (Review):      R1 전체 응답 + 사용자 프롬프트 전달, 상호 검토
R3 (Final):       R2까지 누적 컨텍스트 + 최종 답변 지시
R4~R9 (Extended): 사용자 연장 시 동일 패턴 반복
```

### 3.2 동기화
- 모든 선택된 LLM의 응답 완료 이벤트를 Promise.all 패턴으로 대기
- 한 LLM이라도 미완료 시 다음 라운드 진입 불가
- 타임아웃 도달 시 retry (3~9회 사용자 조정)

### 3.3 라운드 제어
- 기본 3회, 최대 9회, ±1 단위 증감 (마지막 회차에서도 +1 가능)
- 조기 종료: 사용자 명령 시 즉시 종료
- 자동 종료: 지정 회차 완료 시점, 사용자 연장 prompt 표시

## 4. Request Payload Spec
각 LLM에 전달하는 메시지에 포함:
- 사용자 원본 prompt
- 현재 라운드 / 최대 라운드 (예: `R2/3`)
- 타임아웃 (분 단위)
- 직전 라운드 타 LLM 응답 (R2 이후)
- 출력 형식 강제 지시문 (structured, ≤1500 tokens)

## 5. 사용자 개입 정책

| 시점 | 허용 동작 |
|---|---|
| Request 후 ~ 전체 Response 완료 전 | 개입 불가 |
| 라운드 사이 | LLM 추가/제외, 1:1 재질의, 회차 증감, 조기 종료 |
| Response 진행 중 | 다음 회차 prompt 사전 입력 가능 (send buffer만, 전송 차단) |
| MVP 이후 | 선응답 LLM과 개별 "몰래 채팅" |

재질의 결과 → 다음 라운드 컨텍스트에 자동 주입

## 6. 저장 스키마

### 6.1 IndexedDB Object Stores
- `sessions` (id, createdAt, llms[], maxRounds, ...)
- `rounds` (sessionId, roundNo, status, startedAt, completedAt)
- `responses` (sessionId, roundNo, llm, content, tokens, latency)
- `userInputs` (sessionId, roundNo, prompt, sideQueries[])

### 6.2 Export
- 전체 통합 MD 1개 + LLM별 개별 MD N개
- 파일명 컨벤션: `{YYYYMMDD-HHMM}_{sessionId}_{llm}.md`

### 6.3 History 검색
- Index: createdAt(년월일시) / llm / roundNo

## 7. 출력 강제 규칙
- System prompt에 structured output 지시 주입
- 1500 token 상한 (truncation 또는 prompt-level 제한 — 설계 단계에서 결정 필요)
- 장황한 prose 회피 지시

## 8. 비기능 요구
- Chrome 최신 안정 버전
- Windows / macOS / Linux
- 단일 사용자, 암호화 미적용, 모바일 미지원

## 9. 개발 원칙
- **step by step**: 라운드/모듈 단위 점진 구현
- **no guess**: 합의되지 않은 사양 추정 구현 금지 → 명세 확정 후 코드
- **4무 원칙**: 빠짐없이 / 틀림없이 / 다름없이 / 더함없이
- **재사용 자산화**: adapter, round engine, storage layer 템플릿화

## 10. 미결 항목 (확정 필요)
- 1500 token 제한 enforcement 방식 (post-truncate vs prompt-limit)
- LLM adapter의 DOM 변경 대응 전략 (selector 버저닝)
- 재시도 시 idempotency 보장 방식
```

## 📄 파일 3: `30-종합-고객요구.md`

```markdown
# 멀티-LLM 라운드 토론 시스템 — 종합 가이드 (비전문가용 구현 문서)

> 이 문서는 개발 비전문가도 읽고 구현 흐름을 이해할 수 있도록 작성되었습니다.
> 용어는 풀어 쓰고, 각 단계는 "왜" + "어떻게" + "무엇을" 순서로 설명합니다.

---

## 0. 이 시스템이 뭐 하는 건가요?

여러 AI(Claude, ChatGPT, Gemini)에게 **같은 질문을 동시에** 던지고,
그 답들을 **서로 비교/검토하게** 시킨 다음, **최종 결론**까지 도출하는
**크롬 확장 프로그램**입니다.

사람이 회의실에서 전문가 3명을 모아놓고 "각자 의견 내고 → 서로 검토하고 → 최종안 합의" 하는 과정을 AI로 자동화한다고 보면 됩니다.

---

## 1. 어떻게 AI에 접속하나요?

### 1.1 API 키 안 씁니다
보통 AI 개발은 "API 키"라는 비밀번호 같은 걸 발급받아 씁니다.
이 시스템은 **그걸 안 씁니다**. 대신:

- 사용자가 평소처럼 크롬에서 claude.ai, chatgpt.com, gemini.google.com에 **로그인만 해두면**
- 확장 프로그램이 그 **로그인 세션을 그대로 빌려서** AI와 대화합니다

**왜?** API 키는 돈이 들고 관리도 번거롭기 때문입니다.

### 1.2 로그인 자동 탐색
프로그램 실행 시:
1. 3개 AI 사이트에 로그인되어 있는지 확인
2. 로그인 안 된 곳이 있으면 → "로그인하세요" 안내
3. 셋 다 안 되어 있으면 → "기본 AI(Claude) 계정 만드세요" 안내

### 1.3 베이스 코드
`E:\workspace\chathub` 폴더의 **ChatHub**라는 기존 확장 프로그램을 개선해서 만듭니다. 처음부터 만드는 게 아니라 "개조"입니다.

---

## 2. AI를 어떻게 고르나요?

- 로그인 확인된 AI 목록이 표시됩니다
- 사용자가 **2개 이상** 골라야 합니다 (1개는 안 됨 — 토론이 안 되니까요)
- 한번 고른 설정은 **다음에도 기억**됩니다
- 채팅 도중에도 추가/제외 가능 — 단, **라운드와 라운드 사이**에만

---

## 3. 라운드는 뭔가요?

### 3.1 기본 흐름 (3라운드)

```
[사용자가 질문 입력]
        ↓
┌─────────────────────────────────────┐
│ R1: 각 AI가 "혼자서" 답합니다        │
│     (서로 답을 못 봄)                │
└─────────────────────────────────────┘
        ↓ (모든 AI 응답 완료까지 대기)
┌─────────────────────────────────────┐
│ R2: 각 AI에게 "다른 AI들의 R1 답"을  │
│     보여주고 "검토 의견"을 받습니다  │
└─────────────────────────────────────┘
        ↓
┌─────────────────────────────────────┐
│ R3: 각 AI가 "최종 답변"을 냅니다     │
└─────────────────────────────────────┘
        ↓
[사용자가 최종 판단]
```

### 3.2 라운드 수 조정
- 기본 3회 / 최대 9회
- 채팅 도중 +1, -1 가능 (마지막 회차에서도 +1 가능)
- 사용자가 "이만하면 됐다" 싶으면 **조기 종료** 가능
- 정해진 회차 끝나면 자동 종료 → "더 할래요?" 물어봄

### 3.3 동기화
빠른 AI가 먼저 답해도 **느린 AI를 기다립니다**. 모두 답한 후에야 다음 라운드 시작.

---

## 4. 사용자가 도중에 끼어들 수 있나요?

| 상황 | 가능? |
|---|---|
| 라운드 진행 중 (AI들이 답하는 중) | ❌ 끼어들 수 없음 |
| 라운드 사이 | ✅ 가능 |
| 특정 AI와 1:1 재질문 | ✅ 라운드 사이에만 |
| 다음 라운드 질문 미리 입력 | ✅ 가능 (전송은 안 됨, 타이핑만) |
| 특정 AI와 "몰래 채팅" | 🔜 MVP 다음 버전 |

재질문한 내용은 **다음 라운드에 자동으로 반영**됩니다.

---

## 5. 시간 제한

- 기본 **5분** 안에 응답 안 오면 타임아웃
- 사용자가 분 단위로 조정 가능
- 타임아웃 시 재시도 (3~9회, 사용자 설정)
- AI에게 보낼 때 **"제한시간 X분, 지금 R2/3 라운드"** 정보 함께 전달

---

## 6. 저장은 어디에?

### 6.1 자동 저장
- 라운드 끝날 때마다 **브라우저 안의 데이터베이스(IndexedDB)**에 자동 저장
- 서버에 안 올라감 — **내 컴퓨터에만** 있음

### 6.2 파일로 내보내기
- **통합 MD 파일 1개** (전체 대화)
- **AI별 MD 파일** (각 AI 응답만 모음)

### 6.3 이력 검색
- 년/월/일/시 별
- AI 종류별 (Claude만, ChatGPT만 등)
- 라운드별

---

## 7. 응답 형식

- AI가 **장황하게 답 못 하게** 강제합니다
- 응답 1개당 **약 1500 토큰** (한글 약 3000자 정도) 제한
- 구조화된 형식으로 출력 (목록/표 등)

> ⚠️ 1500 토큰 제한은 "AI에게 부탁"으로 할지 "강제 자르기"로 할지 **설계 단계에서 결정 필요**

---

## 8. 안 되는 것들 (제약사항)

- ❌ 크롬 이외 브라우저
- ❌ 모바일 (PC 전용)
- ❌ 여러 사용자 공유 (1인 전용)
- ❌ 서버 저장 (전부 로컬)
- ❌ 데이터 암호화 (안 함)
- ❌ API 키 (안 씀)

**되는 OS**: Windows / macOS / Linux
**브라우저**: Chrome 최신 안정 버전 (Manifest V3 규격)

---

## 9. 개발할 때 지킬 원칙

1. **Step by step** — 한 번에 다 만들지 말고, 단계별로
2. **No guess** — 합의 안 된 건 추측해서 만들지 말기. 물어보고 확정 후 코딩
3. **4무 원칙**
   - 빠짐없이 (요구사항 누락 X)
   - 틀림없이 (오류 X)
   - 다름없이 (요구와 결과 일치)
   - 더함없이 (시키지 않은 것 추가 X)
4. **재사용 자산화** — 다음 프로젝트에서도 쓸 수 있게 템플릿화

---

## 10. 구현 순서 제안 (참고)

```
1단계: ChatHub 코드 분석 → 어떻게 동작하는지 파악
2단계: LLM 3종 로그인 세션 인식 모듈
3단계: 선택 UI + 2개 이상 강제
4단계: 라운드 엔진 (R1 blind → R2 review → R3 final)
5단계: 동기화 + 타임아웃 + 재시도
6단계: IndexedDB 저장
7단계: MD 내보내기 + 이력 검색
8단계: 사용자 개입 (1:1 재질의, 회차 증감)
9단계: 응답 형식 강제 (구조화 + 토큰 제한)
10단계: (MVP 이후) 몰래 채팅
