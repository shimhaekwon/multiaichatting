3사 응답 비교 분석.

**[핵심 공통점]**

3사 모두 동의:
- 100% upfront 설계 = 낭비
- 핵심 영역(아키텍처, 프로토콜, 상태) 깊게
- UI/DB는 가볍게
- 재사용성 중요

**[차이점 - 핵심]**

| 항목 | ChatGPT | Gemini | Claude |
|------|---------|--------|--------|
| 문서 수 | 10개 디렉터리 | **2개로 압축** | 10개 |
| 설계 비율 | 70% 설계 + 30% 보완 | Code-First | 코어 100% / 주변 60% |
| 형식 | 문서 중심 | **코드 자체가 설계서** | 문서 + 템플릿 |
| 특징 | PoC 단계 명시 | TypeScript types.ts 통합 | 결정 체크리스트 |

**[Gemini의 강점 - 인정]**

"문서가 곧 코드" 발상 = 강력함.
- `types.ts` 하나가 DB + API 통합
- 문서 drift 문제 원천 차단
- 재사용 시 복붙 가능

**[ChatGPT의 강점 - 인정]**

"PoC 후 보강" 단계 명시 = 현실적.
- 브라우저 자동화는 실제 DOM 봐야 확정 가능
- 100% upfront 불가능 인정

**[Claude 자체 평가 - 솔직]**

내 안(10문서 + 템플릿)은:
- 안정성 ↑
- 그러나 공수 무거움
- Gemini의 "압축" 관점 부족
- ChatGPT의 "PoC 단계" 관점 부족

**[3사 통합 최적안 - 제안]**

세 답변 합쳐서 새 안:

```
설계 산출물 (3 + α 구조)
─────────────────────────
1. 아키텍처 + 협업 프로토콜 (1문서, 깊이 100%)
   - 시스템 구조
   - 메시지 흐름
   - 라운드 동기화 로직 (의사코드)
   - 상태 머신

2. types.ts (코드 = 설계서)
   - IndexedDB 스키마
   - 메시지 페이로드 타입
   - LLM 어댑터 인터페이스
   - 상태 타입

3. UI/UX 한 장 (와이어프레임 수준)
   - 3-column 레이아웃
   - 상태별 화면 변화

─── PoC 단계 ───
   - 실제 LLM DOM 확인
   - 셀렉터 확정
   - 보완 설계 (어댑터 상세, 에러 케이스)

4. PoC 후 보강 문서 (1개)
   - LLM별 셀렉터 확정본
   - 발견된 엣지 케이스
   - MD 출력 포맷 최종
```

**총 3문서 + PoC + 보강 1문서 = 4문서**

**[공수 비교]**

- Claude 원안: 10문서 → 무거움
- Gemini 안: 2문서 → 가벼우나 PoC 단계 누락
- ChatGPT 안: 10개 디렉터리 → 구조 복잡
- **통합안: 3+1 = 4문서** → 최적

**[재사용성]**

- `types.ts` = 다음 프로젝트 즉시 복붙
- 아키텍처 문서 = 템플릿화
- PoC 패턴 = 프로세스 가이드로 자산화

**[권장 진행 순서]**

1. 문서1 (아키텍처+프로토콜) 작성
2. 문서2 (types.ts) 작성
3. 문서3 (UI 한 장) 작성
4. PoC 진행 (실제 LLM 사이트 확인)
5. 문서4 (보강) 작성
6. 본 구현

**[결정 요청]**

- 통합안(4문서 + PoC) OK?
- 또는 다른 조합 선호?

(이 통합 결과 자체도 3사에 다시 전달 검토 가능. 단, 반복 시 수렴 vs 시간 소모 트레이드오프 발생.)
