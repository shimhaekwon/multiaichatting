225. UI/UX 설계 — ai-sarangbang

자기완결. [221] 제품·규칙 · [222] coordinator · [223] 데이터모델 · [224] 아키텍처를 화면으로 구체화. 90s PC통신(하이텔·나우누리) 감성의 "사랑방". 본 문서가 [221] O1(90s 감성 구체 스펙)을 확정한다.
변경 이력: v1 (2026-06-02 초안) · 충돌 시 규칙은 [221] §3 우선 · 표기: 이모지 미사용.

0. 디자인 원칙

  • 터미널이 곧 방 — 하나의 어두운 모노스페이스 화면. 창·탭·사이드바 최소화. 대화 로그가 주인공.
  • 화자는 색과 대괄호로 구분 — [이름] 발언. 사람·AI마다 고정 색([223] Participant.color).
  • 한 번에 한 명(floor, [221] R2) — "지금 누가 말하는가"를 roster 하이라이트 + 타이핑 커서로 즉시 인지.
  • 글자별 타이핑 — 90s 감성의 핵심. 턴제 규칙이 눈에 보이게 만드는 장치(한 명 다 찍힌 뒤 다음).
  • 귓속말은 시각적으로 격리([221] R4) — 공개 로그와 다른 색·영역. 저장 안 됨을 UI가 명시.
  • 접근성 — 색만으로 구분하지 않음(이름 라벨 병행), 명도 대비 확보, 모션 민감자 위해 타이핑 off 토글.

5. 인터랙션 상태 (turn-state 시각화, [222] §3)

상태로그 표현Roster 표현
queued 대기(아직 발언 없음)이름 옆 "대기" (파랑)
speaking 발언중(floor)[이름] 본문…█ 타이핑+커서행 하이라이트 + "발언중"(노랑)
done 완료[이름] 본문 정적"완료"(초록)
stopped 중단[이름] 부분본문 (중단됨) 빨강 표식·부분 보존"중단"(빨강)
error 오류 (Message 전용 status — turn-state 아님, [222] §3 H1)[이름] (응답 오류)Participant는 중단(stopped)으로 표시

[H1] `error`는 그 발언(Message)의 status일 뿐이고, 참가자(Participant) turn-state는 idle/queued/speaking/done/stopped 5종뿐. 드라이버 에러 시 Message=error·Participant=stopped([222] §9).

바지인(D3): 사람이 발언중인 AI를 끊으면 그 줄이 (중단됨)으로 고정(부분 텍스트 보존, [222] §5 completedNaturally 가드) → 사람 발언이 새 줄로.

6. 귓속말(whisper) UI ([221] R4 · [222] §7)

  • 진입: roster의 참가자 클릭 → "[귓속말]" 또는 입력창에 /w 이름 내용.
  • 격리 표현: 보라색(--whisper) 별도 오버레이/드로어. 공개 로그와 절대 섞이지 않음.
  • 휘발 명시: 패널 상단에 "이 대화는 저장되지 않습니다" 고정 배너. 닫으면 사라짐(MD·스냅샷 미기록, [223] §2).
  • 턴 진행 중에도 가능(floor 독립). 같은 모델 경합 시 잠깐 대기 가능(P1, [222] C2).

7. 반응형 · 접근성

  • 좁은 화면(≤680px): roster 접힘 → 상단 요약 바 또는 토글. 로그·입력 우선.
  • 색약: 색 + 이름 라벨 병행(색 단독 의존 금지). 명도 대비 WCAG AA 목표.
  • 모션 민감: 타이핑 off 토글(config.typingMs=0) → 즉시 표시. prefers-reduced-motion 존중.
  • 키보드: Enter 전송, Esc 귓속말 닫기, ↑ 직전 입력 복구(선택).
  • i18n: 모든 라벨 ko/en(한글 하드코딩 금지, [221] §6). 시스템 문구도 사전 키.
