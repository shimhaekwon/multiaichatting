### 1. 입출력 시간 차(Latency) 극복을 위한 UI/UX 제어

각 모델마다 첫 토큰이 나오는 속도(Time to First Token)와 전체 답변이 완료되는 시간이 완전히 다릅니다. (예: DeepSeek이나 Gemini가 속도를 내는 동안, Claude는 조금 더 묵직하게 답변을 출력할 수 있죠.) 이를 제어하기 위한 세 가지 장치가 필요합니다.

* **독립적인 개별 스트리밍 UI (독립 큐 관리)**
    * 전체 화면을 하나로 묶어 '모든 모델의 답변이 끝날 때까지 대기'하게 만들면 UX가 매우 답답해집니다.
    * 각 모델의 출력 영역에 **독립적인 스피너(로딩 애니메이션)와 스톱 버튼**을 개별 배치해야 합니다. 먼저 끝난 모델의 답변은 사용자가 즉시 읽을 수 있도록 하고, 느린 모델은 백그라운드에서 계속 스트리밍되도록 둡니다.
* **글로벌 입력창 '잠금(Disable)' 정책 정의**
    * **옵션 A (엄격한 제어):** 선택된 모든 모델 중 *단 하나라도 답변을 생성 중*이라면 아래 메인 입력창을 비활성화(Disable)하고 '생성 중...' 상태로 둡니다. 대화의 컨텍스트(순서)가 꼬이는 것을 막는 가장 안전한 방법입니다.
    * **옵션 B (느슨한 제어):** 먼저 끝난 모델이 있다면 입력창을 열어주되, 아직 답변 중인 모델에게는 "이전 질문에 대한 답변이 끝나면 다음 질문을 처리해라"는 식으로 요청 큐(Queue)에 쌓아두는 방식입니다. (구현 난이도가 높습니다.)

### 2. 기술적 제어 및 아키텍처 (개발자 관점)

만약 이 UI를 직접 프론트엔드(Vue3/Nuxt3 등)로 프로토타이핑하거나, 에이전트 환경에서 제어하신다면 아래 구조가 효율적입니다.

* **Promise.all() vs Promise.allSettled()**
    * 여러 API를 동시에 호출할 때 `Promise.all`을 쓰면 한 모델에서 에러가 났을 때 전체가 터져버립니다.
    * 반드시 **`Promise.allSettled()`**를 사용하거나, 각 모델의 API 호출 함수 내부에서 `try-catch`로 에러를 완벽히 격리해야 합니다. "Claude는 성공했지만, Gemini는 API 할당량 초과로 실패"한 상황을 UI에 각각 독립적으로 그려주어야 하기 때문입니다.
* **SSE (Server-Sent Events) 및 웹소켓 핸들링**
    * 텍스트가 주르륵 나오는 스트리밍을 구현할 때, 프론트엔드 상태 관리(State)에서 각 체크된 모델의 ID를 Key로 가지는 객체(예: `{ claude: "답변중...", gemini: "완료" }`)를 만들어 실시간으로 화면을 렌더링해야 동기화가 깨지지 않습니다.

* **삼자 대화 버퍼(Buffer)의 부재:**
  서로 대화를 주고받게 하려면 `Claude 답변 완료 이벤트 발생 -> 그 내용을 복사해서 Gemini 입력창에 바인딩 -> Gemini 전송`이라는 흐름 제어(Flow Control)가 필요한데, 이 프로그램의 '사용자 계정 모드'는 단순 브로드캐스팅(동시 살포)만 하도록 설계되어 있습니다.

## 💡 구상하신 방식의 단계별 확장 프로세스

### 1단계: 멀티캐스팅 (One-to-Many Request)
사용자가 질문(Prompt)을 하나 던지면, 켜져 있는 모든 LLM(ChatGPT, Claude, Gemini, DeepSeek 등)에 **동시에** 요청을 찌릅니다.
* **출력 방식:** 화면에 `[ChatGPT]: 결과`, `[Claude]: 결과` 형태로 멀티 윈도우 UI를 구성해 한눈에 비교합니다. (현재 챗올(ChatALL) 같은 도구들이 이 단계를 구현하고 있습니다.)

### 2단계: 크로스 피드백 & 토론 (Multi-Agent Debate)
대박인 건 바로 이 지점입니다. **"그 결과창 자체를 다시 각각 LLM들에게 전달한다"**는 아이디어인데요. 이걸 개발 쪽에서는 다음과 같이 파이프라인을 짭니다.

1. **상호 교차 검증 (Cross-Evaluation):**
   * ChatGPT가 낸 답변을 Claude에게 주고 *"ChatGPT의 답변 중 오류나 보완할 점을 지적해줘"*라고 요청합니다.
   * 반대로 Claude의 답변을 Gemini에게 검증받습니다.
2. **합의 형성 (Consensus/Synthesis):**
   * 마지막에 **'메인 에이전트(예: 가장 추론 능력이 뛰어난 모델)'**를 하나 두고, 앞서 나온 모든 LLM들의 답변과 상호 비판 내용을 한데 모아서 최종 프롬프트로 입력합니다.
   * *"위의 다양한 LLM들의 의견을 종합해서, 가장 객관적이고 완벽한 최종 코딩/답변을 도출해줘."*

## 🛠️ 이 방식의 엄청난 장점과 현실적인 걸림돌

### 👍 장점: 환각(Hallucination)의 극적인 감소
각 LLM마다 강점이 다릅니다. (예: 코딩은 Claude, 가성비와 빠른 추론은 DeepSeek, 최신 정보 검색은 Perplexity 등) 이들의 집단지성을 이용하면 혼자서 답변할 때 발생하는 **환각 현상이나 논리적 오류를 서로 필터링**해 주기 때문에 결과물의 퀄리티가 비약적으로 상승합니다.

### 👎 단점: 브라우저 세션 방식의 한계 (속도와 차단)
앞서 보신 '크롬 계정 연동 방식'으로 이 토론 시스템을 확장하려면 현실적인 제약이 걸립니다.
* **컨텍스트 창(Context Window) 압박:** 각 LLM의 결과물 전체를 다시 프롬프트로 말아 넣으면 텍스트 양이 급증하는데, 브라우저 매크로 방식은 이를 처리하기에 속도가 느립니다.
* **레이트 리밋(Rate Limit) 및 봇 차단:** 브라우저 세션을 통해 단시간에 수많은 요청과 대용량 텍스트가 오가면, 각 서비스(특히 보안이 까다로운 OpenAI나 Anthropic)에서 *'매크로/봇 사용'*으로 감지해 일시적으로 계정을 차단하거나 캡차(CAPTCHA)를 띄울 확률이 높습니다.

## 🚀 현업/오픈소스에서는 어떻게 구현할까?

Deeps님이 생각하신 이 강력한 워크플로우를 안정적으로 돌리기 위해 보통 현업이나 헤비 유저들은 **공식 API 기반의 오케스트레이션 툴**을 사용합니다.

* **LangGraph / CrewAI:** 에이전트 여러 개를 두고 서로 대화하게 만드는 프레임워크입니다. "A가 초안 잡고, B가 검수하고, C가 최종 빌드해"라는 구조를 코드로 짤 수 있습니다.
* **OpenWebUI + Pipelines:** UI단에서 여러 LLM의 답변을 합성(Synthesis)하는 파이프라인을 붙여서 사용합니다.

브라우저 세션 연동은 '개인용 무료 테스트 및 가벼운 비교'에는 최고지만, 구상하신 **[결과 취합 ➡️ 다시 피드백 ➡️ 최종본 도출]**이라는 고도화된 루프를 돌리기 시작하면 결국 공식 API(또는 로컬 LLM 서버)를 엮어서 파이프라인을 구축하는 방향으로 진화하게 됩니다. 

생각하신 아키텍처 방향성이 정확히 요즘 AI 엔지니어링이 나아가는 방향과 일치하네요. 혹시 이 구조를 활용해서 현재 진행 중인 프로젝트의 문서 자동화나 코드 검수 파이프라인에 구체적으로 적용해 보고 싶으신 부분이 있으신가요?

## 📊 3 라운드 제한 시스템의 흐름 예시

구상하신 대로 "모든 모델의 응답이 끝나야 다음 단계로 넘어간다"는 룰을 적용하면 다음과 같은 깔끔한 3단계 파이프라인이 완성됩니다.

```
[사용자 요구사항 입력]
      │
      ▼
【Round 1: 초안 작성 및 관점 제시】
 ├─ ChatGPT: "저는 A 방식으로 구현하겠습니다."
 ├─ Claude:  "저는 B 방식으로 구현하겠습니다."
 └─ Gemini:  "저는 C 방식으로 구현하겠습니다."
      │ (모든 응답 수집 완료 ➡️ 다음 라운드로 일괄 전달)
      ▼
【Round 2: 교차 검증 및 상호 비판】
 ├─ ChatGPT: "Claude의 B 방식은 보안 취약점이 있고, Gemini의 C는 과합니다."
 ├─ Claude:  "ChatGPT의 A 방식은 확장성이 떨어집니다."
 └─ Gemini:  "A와 B의 장점을 합친 하이브리드가 좋아 보입니다."
      │ (모든 응답 수집 완료 ➡️ 최종 라운드로 일괄 전달)
      ▼
【Round 3: 최종 타협 및 확답 도출】
 └─ 각 LLM이 앞선 토론을 바탕으로 '최종 정제된 정답'을 한 줄로 요약 제출.
```

---

## 👍 이 방식이 컨텍스트 폭발을 막는 이유

1. **컨텍스트의 선형적(Linear) 통제:** 무제한 토론을 시키면 대화가 핑퐁되면서 컨텍스트가 기하급수적으로 늘어나지만, 딱 3번으로 제한하면 전체 텍스트 소모량이 `[모델 수 × 3]`으로 고정되므로 브라우저 메모리나 토큰 한계(Context Window)를 넘길 일이 거의 없습니다.
2. **동기식(Synchronous) 제어의 안정성:**
   하나의 LLM이 혼자 흥분해서 연속으로 답변을 쏟아내지 못하게 락(Lock)을 걸어두었기 때문에, 브라우저 세션이 한꺼번에 수많은 요청을 받아 먹통이 되는(Rate Limit) 현상을 물리적으로 방지할 수 있습니다.

## 🛠️ 성공적인 구현을 위한 꿀팁 (프롬프트 엔지니어링)

브라우저 기반 매크로 환경에서 이 3라운드 시스템이 엇나가지 않고 칼같이 작동하게 하려면, 각 라운드를 넘길 때 **"역할 정의 프롬프트"**를 명확하게 주입해야 합니다. LLM들은 놔두면 자꾸 서론/결론을 길게 빼서 텍스트를 낭비하기 때문입니다.

* **Round 1 전달 시:** > "너는 3인의 전문가 토론단 중 한 명이야. 핵심 요구사항에 대해 **[요약본 3줄]** 이내로 너의 독자적인 해결책만 제시해. 다른 말은 붙이지 마."
* **Round 2 전달 시:** > "여기에 다른 전문가들의 의견이 모였어. **[다른 의견의 취약점 1가지]**와 **[내 의견의 보완점 1가지]**만 아주 간결하게 반박해."
* **Round 3 전달 시:**
  > "치열한 토론이 끝났어. 앞선 내용을 종합해서 사용자가 바로 복사해서 쓸 수 있는 **[최종 확답/코드 블록]**만 출력하고 대화를 종료해."

## ⚠️ 브라우저 세션 환경에서의 마지막 복병: '속도(Latency)'

구조적으로는 완벽한데, 실제로 돌려보시면 **'속도'**에서 아주 약간의 답답함이 생길 수 있습니다. 
"모든 LLM의 응답이 끝나야 다음으로 넘어간다"는 룰 때문에, 응답 속도가 가장 느린 LLM(예: 생각(Thinking) 모델이나 브라우저 응답이 지연되는 모델)이 전체 파이프라인의 병목(Bottleneck)이 됩니다.

따라서 구현하실 때 **"특정 모델이 15초 동안 응답이 없으면 해당 라운드에서 해당 모델은 제외(Timeout)하고 다음 라운드로 진행"**하는 예외 처리 로직만 살짝 얹어주시면 아주 훌륭한 나만의 '집단지성 AI 툴'이 될 것 같습니다.

Deeps님이 말씀하신 **"사용자 최종 판단 + 근거 추적 + 3단계 Step-by-Step"** 구조는 AI 아키텍처에서 말하는 이른바 **'HITL (Human-in-the-Loop, 인간 참여형)'** 시스템의 아주 훌륭한 예시입니다. 

이 개념을 입혀서 실제 사용자가 마주할 전체 워크플로우를 짜보면 정말 매끄러운 내비게이션이 나옵니다.

## 🗺️ 사용자 중심의 3단계 Step-by-Step 워크플로우

### 1단계 : 발산 (Multi-Perspective Idea)
* **AI들의 역할:** 사용자의 요구사항을 받아서 각 LLM이 독자적인 관점(A안, B안, C안)과 솔루션을 3 response 이내로 간결하게 던집니다.
* **화면 노출:** 사용자는 대시보드에서 각 AI들이 제안한 초안을 한눈에 비교합니다.

### 2단계 : 교차 검증 및 근거 제시 (Cross-Debate & Grounding)
* **AI들의 역할:** 1단계 결과물을 바탕으로 서로의 약점을 찌르고(상호 비판), 동시에 **"왜 내 방식이 맞는지"에 대한 구체적인 판단 근거**를 리포트 형태로 출력합니다.
* **사용자의 개입 (HITL):** 여기서 사용자가 개입합니다. 결과창을 보고 *"A 모델, 네가 말한 근거 중에서 이 라이브러리를 쓰면 버전 호환성 문제는 없어?"* 혹은 *"B 모델, 네가 제시한 근거의 비용 시뮬레이션을 더 자세히 보여줘"*라며 **특정 모델의 근거를 다시 심문(Deep Dive)**할 수 있습니다.

### 3단계 : 최종 수렴 및 확정 (Human Decision & Final Output)
* **사용자의 역할:** AI들의 근거와 상호 반박을 모두 확인한 사용자가 "오케이, 이번 프로젝트는 Claude의 확장성과 ChatGPT의 예외처리 방식을 섞어서 간다!"라고 **최종 판단**을 내립니다.
* **AI들의 역할:** 사용자의 최종 결정을 입력받은 메인 에이전트가 그 선택에 맞춰 실제 구현해야 할 완벽한 코드, 혹은 최종 기획서 아웃풋을 뽑아내며 프로세스를 종료합니다.

## 🎯 이 구조가 주는 진짜 가치

1. **블랙박스(Black Box) 해소:** AI가 "이게 정답입니다"라고 던지는 결과물은 왜 그렇게 나왔는지 알 수 없어 찝찝할 때가 많습니다. 하지만 이 구조는 **판단의 근거(Rationale)를 강제로 출력**하게 만들고 사용자가 이를 추적할 수 있기 때문에, 결과물의 신뢰성이 극대화됩니다.

3. **의사결정 시간 단축:** PM이나 Technical Lead 입장에서 혼자 하면 서너 시간 걸릴 '아키텍처 비교 분석 및 리스크 검토'를, AI 참모들이 1~2분 만에 끝내주니 최종 결정만 내리면 됩니다.

# Multi-LLM Ensemble & Debate Platform (Deeps Architecture v1.0)
## 시스템 설계서 (System Architecture Specification)

## 1. 아키텍처 개요 & 핵심 메커니즘
본 시스템은 Chrome 브라우저 세션(Cookie/Session) 공유 기반의 다중 LLM 호출 구조 체제에서 발생하는 **컨텍스트 폭발, 레이트 리밋(Rate Limit), 브라우저 행(Hang)** 현상을 물리적으로 제어하기 위해 **'동기식 3라운드 라운드 로빈(Synchronous 3-Round Round-Robin)'** 및 **'HITL(Human-in-the-Loop)'** 가이드라인을 강제 적용하는 오케스트레이션 플랫폼입니다.

```
                  [ 사용자 Prompt 입력 ]
                             │
                             ▼
 ┌────────────────────────────────────────────────────────┐
 │ 1단계 (발산): 개별 LLM 초안 도출 (Max 3 Responses 락)    │
 └───────────────────────────┬────────────────────────────┘
                             ▼ (모든 LLM 응답 수집 완료 대기 - 동기식 블로킹)
 ┌────────────────────────────────────────────────────────┐
 │ 2단계 (토론): 타 모델 결과 취합 후 상호 교차 검증/근거 제출 │
 └───────────────────────────┬────────────────────────────┘
                             ▼
               [ HITL : 사용자 개입 / 근거 추적 심문 ] ◀─── 사용자가 추가 질문/보완 가능
                             │
                             ▼
 ┌────────────────────────────────────────────────────────┐
 │ 3단계 (수렴): 사용자가 선택한 최적안 기준 최종 결과 빌드  │
 └────────────────────────────────────────────────────────┘
```

## 3. API 및 데이터 흐름 설계 (Data Flow & Architecture)

본 시스템은 공식 API가 아닌 브라우저 내장 세션(인젝트 스크립트/확장 프로그램 백그라운드)을 제어하므로, **동기식 상태 관리 프레임워크**가 핵심입니다.

### 3.1 내부 통신 흐름 (Sequence Diagram)
1. **User Request:** 사용자가 UI에 Prompt 입력 및 전송 버튼 클릭.
2. **Orchestrator Orchestration:** * 입력된 오리지널 프롬프트에 `[Round 1 가이드 프롬프트]`를 인터셉트하여 마스킹 결합.
   * 각 LLM 인젝션 스레드(Worker) 호를 웹소켓/메시지 버스로 호출.
3. **Synchronization Barrier (동기식 락):**
   * 각 Worker들로부터 `RESPONSE_END` 이벤트가 모두 수집되거나, 지정된 `Timeout(15초)`이 만료될 때까지 메인 상태 머신을 `BLOCK` 상태로 유지.
4. **Context Injection & Round 2 Trigger:**
   * 수집된 $N$개의 답변을 하나의 컨텍스트 텍스트(`[LLM A]: ... [LLM B]: ...`)로 합성.
   * `[Round 2 상호 비판 프롬프트]` 사양과 결합하여 다시 각 에이전트 탭으로 Payload 재전송.

### 3.2 핵심 API 스펙 (Back-end / Extension Background Context)

#### ① 글로벌 요청 세션 개시 (POST `/api/v1/ensemble/session`)
* **Request Payload:**
```json
{
  "sessionId": "deeps-session-20260526",
  "userPrompt": "사용자 요구사항 원본 텍스트",
  "activeAgents": ["chatgpt", "claude", "gemini"],
  "config": {
    "maxRounds": 3,
    "timeoutMs": 15000
  }
}
```
* **Response:** `200 OK` (세션 초기화 및 상태 머신 대기 상태 진입)

#### ② 에이전트별 스트리밍 상태 수집 (WS `/ws/ensemble/stream`)
* **Message Payload (Agent ➡️ Orchestrator):**
```json
{
  "sessionId": "deeps-session-20260526",
  "agentId": "claude",
  "currentRound": 1,
  "status": "STREAMING",
  "textDelta": "구현 핵심 내용...",
  "isFinal": false
}
```

#### ③ 라운드 동기화 상태 제어 및 수렴 락 해제 (GET `/api/v1/ensemble/session/{id}/status`)
* **Response Payload:**
```json
{
  "sessionId": "deeps-session-20260526",
  "currentRound": 2,
  "isBarrierCleared": true, 
  "capturedResponses": {
    "chatgpt": "ChatGPT의 Round 2 텍스트",
    "claude": "Claude의 Round 2 텍스트"
  }
}
```

## 4. 예외 처리 및 안정성 고도화 설계 (Anti-Blocking Strategy)

브라우저 계정 매크로 방식의 치명적인 취약점인 **'차단(Rate Limit)'** 및 **'교착 상태(Deadlock)'**를 방지하기 위한 기술적 장치입니다.

1. **타임아웃 배리어 탈출 메커니즘 (Timeout Drop-out):**
   * 한 개의 AI가 로딩 지연 등으로 인해 15초 동안 응답이 없으면, 오케스트레이터는 해당 에이전트 스레드를 즉시 `DROP_ROUND` 처리하고, 나머지 정상 응답한 모델들의 결과물만 묶어서 즉시 다음 라운드로 강제 진행시킵니다.
2. **토큰 절약을 위한 프롬프트 다이어트 가이드라인:**
   * 각 라운드가 넘어갈 때 이전 라운드의 전체 마크다운 포맷을 그대로 넘기면 텍스트가 폭발합니다. 시스템 백그라운드에서 코드 블록(` ``` `)과 불필요한 서론/결론 컴포넌트를 정규식으로 자동 정제(Trimming)한 후 **순수 핵심 아키텍처 관점 텍스트만 파싱하여 다음 라운드의 컨텍스트로 주입**합니다.
3. **가상 입력 및 휴먼 딜레이 인젝션 (Anti-Bot Bypass):**
   * 브라우저 세션 차단을 방지하기 위해 각 라운드 페이로드 전송 시 0.5초~1.2초 사이의 가상 무작위 지연(Random Latency) 및 인풋 에뮬레이션을 적용하여 보안 봇(Bot Detection) 우회 메커니즘을 상시 가동합니다.

이 설계서는 기술적 제약(용량 한계 및 속도 병목)을 인간 중심의 인터랙션 구조(Step-by-Step 가이드라인 및 라운드 블로킹)로 승화시킨 완벽한 빌드 가이드라인입니다. 본 도면 그대로 인젝션 스크립트 파이프라인 및 프런트엔드 그리드 대시보드 개발을 추진하시면 됩니다.

플랫폼 연동의 한계를 '3라운드 동기식 제약'과 '인간 참여형(HITL) 루프'로 해결하는 독창적인 아키텍처입니다. 제안하신 사양을 바탕으로, 즉시 개발에 착수할 수 있는 수준의 **실질적인 시스템 설계서(System Architecture Specification)**를 작성했습니다.

# Multi-LLM Ensemble & Debate Platform (Deeps Architecture v1.0)
## 시스템 설계서 (System Architecture Specification)

## 1. 아키텍처 개요 & 핵심 메커니즘
본 시스템은 Chrome 브라우저 세션(Cookie/Session) 공유 기반의 다중 LLM 호출 구조 체제에서 발생하는 **컨텍스트 폭발, 레이트 리밋(Rate Limit), 브라우저 행(Hang)** 현상을 물리적으로 제어하기 위해 **'동기식 3라운드 라운드 로빈(Synchronous 3-Round Round-Robin)'** 및 **'HITL(Human-in-the-Loop)'** 가이드라인을 강제 적용하는 오케스트레이션 플랫폼입니다.

```
                  [ 사용자 Prompt 입력 ]
                             │
                             ▼
 ┌────────────────────────────────────────────────────────┐
 │ 1단계 (발산): 개별 LLM 초안 도출 (Max 3 Responses 락)    │
 └───────────────────────────┬────────────────────────────┘
                             ▼ (모든 LLM 응답 수집 완료 대기 - 동기식 블로킹)
 ┌────────────────────────────────────────────────────────┐
 │ 2단계 (토론): 타 모델 결과 취합 후 상호 교차 검증/근거 제출 │
 └───────────────────────────┬────────────────────────────┘
                             ▼
               [ HITL : 사용자 개입 / 근거 추적 심문 ] ◀─── 사용자가 추가 질문/보완 가능
                             │
                             ▼
 ┌────────────────────────────────────────────────────────┐
 │ 3단계 (수렴): 사용자가 선택한 최적안 기준 최종 결과 빌드  │
 └────────────────────────────────────────────────────────┘
```

## 3. API 및 데이터 흐름 설계 (Data Flow & Architecture)

본 시스템은 공식 API가 아닌 브라우저 내장 세션(인젝트 스크립트/확장 프로그램 백그라운드)을 제어하므로, **동기식 상태 관리 프레임워크**가 핵심입니다.

### 3.1 내부 통신 흐름 (Sequence Diagram)
1. **User Request:** 사용자가 UI에 Prompt 입력 및 전송 버튼 클릭.
2. **Orchestrator Orchestration:** * 입력된 오리지널 프롬프트에 `[Round 1 가이드 프롬프트]`를 인터셉트하여 마스킹 결합.
   * 각 LLM 인젝션 스레드(Worker) 호를 웹소켓/메시지 버스로 호출.
3. **Synchronization Barrier (동기식 락):**
   * 각 Worker들로부터 `RESPONSE_END` 이벤트가 모두 수집되거나, 지정된 `Timeout(15초)`이 만료될 때까지 메인 상태 머신을 `BLOCK` 상태로 유지.
4. **Context Injection & Round 2 Trigger:**
   * 수집된 $N$개의 답변을 하나의 컨텍스트 텍스트(`[LLM A]: ... [LLM B]: ...`)로 합성.
   * `[Round 2 상호 비판 프롬프트]` 사양과 결합하여 다시 각 에이전트 탭으로 Payload 재전송.

### 3.2 핵심 API 스펙 (Back-end / Extension Background Context)

#### ① 글로벌 요청 세션 개시 (POST `/api/v1/ensemble/session`)
* **Request Payload:**
```json
{
  "sessionId": "deeps-session-20260526",
  "userPrompt": "사용자 요구사항 원본 텍스트",
  "activeAgents": ["chatgpt", "claude", "gemini"],
  "config": {
    "maxRounds": 3,
    "timeoutMs": 15000
  }
}
```
* **Response:** `200 OK` (세션 초기화 및 상태 머신 대기 상태 진입)

#### ② 에이전트별 스트리밍 상태 수집 (WS `/ws/ensemble/stream`)
* **Message Payload (Agent ➡️ Orchestrator):**
```json
{
  "sessionId": "deeps-session-20260526",
  "agentId": "claude",
  "currentRound": 1,
  "status": "STREAMING",
  "textDelta": "구현 핵심 내용...",
  "isFinal": false
}
```

#### ③ 라운드 동기화 상태 제어 및 수렴 락 해제 (GET `/api/v1/ensemble/session/{id}/status`)
* **Response Payload:**
```json
{
  "sessionId": "deeps-session-20260526",
  "currentRound": 2,
  "isBarrierCleared": true, 
  "capturedResponses": {
    "chatgpt": "ChatGPT의 Round 2 텍스트",
    "claude": "Claude의 Round 2 텍스트"
  }
}
```

## 4. 예외 처리 및 안정성 고도화 설계 (Anti-Blocking Strategy)

브라우저 계정 매크로 방식의 치명적인 취약점인 **'차단(Rate Limit)'** 및 **'교착 상태(Deadlock)'**를 방지하기 위한 기술적 장치입니다.

1. **타임아웃 배리어 탈출 메커니즘 (Timeout Drop-out):**
   * 한 개의 AI가 로딩 지연 등으로 인해 15초 동안 응답이 없으면, 오케스트레이터는 해당 에이전트 스레드를 즉시 `DROP_ROUND` 처리하고, 나머지 정상 응답한 모델들의 결과물만 묶어서 즉시 다음 라운드로 강제 진행시킵니다.
2. **토큰 절약을 위한 프롬프트 다이어트 가이드라인:**
   * 각 라운드가 넘어갈 때 이전 라운드의 전체 마크다운 포맷을 그대로 넘기면 텍스트가 폭발합니다. 시스템 백그라운드에서 코드 블록(` ``` `)과 불필요한 서론/결론 컴포넌트를 정규식으로 자동 정제(Trimming)한 후 **순수 핵심 아키텍처 관점 텍스트만 파싱하여 다음 라운드의 컨텍스트로 주입**합니다.
3. **가상 입력 및 휴먼 딜레이 인젝션 (Anti-Bot Bypass):**
   * 브라우저 세션 차단을 방지하기 위해 각 라운드 페이로드 전송 시 0.5초~1.2초 사이의 가상 무작위 지연(Random Latency) 및 인풋 에뮬레이션을 적용하여 보안 봇(Bot Detection) 우회 메커니즘을 상시 가동합니다.

이 설계서는 기술적 제약(용량 한계 및 속도 병목)을 인간 중심의 인터랙션 구조(Step-by-Step 가이드라인 및 라운드 블로킹)로 승화시킨 완벽한 빌드 가이드라인입니다. 본 도면 그대로 인젝션 스크립트 파이프라인 및 프런트엔드 그리드 대시보드 개발을 추진하시면 됩니다.
**bard**: 이전 대화에서 정리된 설계서는 Deeps님이 구상하신 핵심 메커니즘을 훌륭하게 컴포넌트화했습니다. PL이자 아키텍트 관점에서 이 설계서가 **"실행 가능한(Actionable) 실제 구현 도면"**이 되기 위해 검토한 결과, 구조는 매우 탄탄하나 **브라우저 세션(웹 확장 프로그램) 환경의 물리적 특성상 반드시 보완해야 할 치명적인 리스크와 아키텍처적 간극**이 몇 가지 발견되었습니다. 

실제 개발팀에 이 스펙을 넘기기 전에 반영해야 할 **PM/PL 관점의 코드레벨 기술 검토 및 보완점**을 공유해 드립니다.

## 🔍 [PL 관점] 핵심 기술 검토 및 보완 요구사항

### 1. 백엔드 API 아키텍처의 현실성 보완 (Extension 구조의 한계)
* **문제점:** 설계서 3.2절을 보면 `POST /api/v1/ensemble/session` 같은 전통적인 백엔드 API와 백엔드 중심의 웹소켓 구조로 설계되어 있습니다. 하지만 이 시스템은 **사용자 크롬 브라우저의 쿠키와 세션**을 빌려 써야 하므로, 외부 독립 백엔드 서버가 직접 각 LLM(OpenAI 등)에 요청을 찌를 수 없습니다. (그렇게 하면 백엔드 IP가 차단되거나 세션 인증이 풀립니다.)
* **보완 가이드:** 메인 오케스트레이터(컨트롤 타워)는 백엔드가 아니라 **Chrome Extension의 `Background Script (Service Worker)`**가 되어야 합니다. 
  * 백엔드 API 명세로 표현된 부분은 Extension 내부의 `Message Passing API (chrome.runtime.sendMessage)` 구조로 전환되어야 개발팀이 삽질하지 않습니다.

### 2. 동기식 배리어(Synchronization Barrier)의 상태 관리 교착 리스크
* **문제점:** Step 2(토론)로 넘어가기 위해 모든 에이전트의 `RESPONSE_END`를 기다리는 동기식 블로킹(Blocking)을 채택했습니다. 만약 특정 LLM 탭이 렌더링에 실패하거나 크래시가 나면 시스템 전체가 먹통이 될 위험이 큽니다.
* **보완 가이드:** 15초 타임아웃 외에, 각 에이전트의 상태를 관리하는 **`Circuit Breaker(서킷 브레이커)` 패턴**을 확장 프로그램단에 명시해야 합니다. 응답이 없는 탭은 즉시 무시하고 `Promise.allSettled()` 형태로 래핑하여 실패한 에이전트의 컨텍스트는 `[LLM Name: 응답 실패/제외됨]`으로 가공해 다음 단계로 무조건 넘겨주는 방어 코드가 설계에 포함되어야 합니다.

### 3. 가장 중요한 보완: 프롬프트 주입(Injection) 시점의 '컨텍스트 오염' 방지
* **문제점:** Step 2에서 타 모델들의 결과를 취합해 각 LLM에게 다시 전달할 때, 각 LLM 탭은 **이전 Step 1의 대화 기록(Context)을 이미 브라우저 세션상에 보관**하고 있습니다. 
* **보완 가이드:** * 만약 기존 대화창에 그대로 *"자, 이게 다른 LLM들의 의견이야"*라고 던지면, 해당 LLM은 자신이 Step 1에서 했던 말과 타인의 말을 헷갈리는 **컨텍스트 오염(Context Contamination)**이 발생합니다.
  * 따라서, 매 라운드마다 **"완전히 새로운 시크릿 대화 세션(New Thread)"을 백그라운드에서 강제로 개설**하여 프롬프트를 주입할 것인지, 아니면 **기존 스레드 내에서 완벽하게 격리된 구분자(Delimiter)**를 쓸 것인지 설계서에 명시해 주어야 프롬프트 엔지니어링이 꼬이지 않습니다.

### 3. API 및 데이터 흐름 설계 (Extension 구조로 수정)

#### 3.1 내부 통신 흐름 (Chrome Extension Architecture)
본 시스템은 외부 백엔드가 아닌 Chrome Extension의 `Service Worker`를 중심으로 브라우저 탭(Content Script)들과 메시지를 주고받는 구조를 가집니다.

```
[사용자 UI (Popup/Dashboard)]       [Service Worker (Orchestrator)]      [개별 LLM Tabs (Content Script)]
          │                                      │                                      │
          ├────────── userPrompt 전송 ──────────>│                                      │
          │                                      ├─────── Round 1 프롬프트 인젝션 ──────>│
          │                                      │                                      ├─ (추론 및 스트리밍)
          │                                      |<─────── 응답 델타 및 완료 시그널 ──────┤
          │                                      │   (Barrier 완료 혹은 15s 타임아웃)      │
          │                                      │                                      │
          │                                      ├─ 텍스트 정제 후 Round 2 토론문 주입 ──>│
          │                                      │                                      ├─ (상호 비판 수행)
          │<───────── 데이터 레이아웃 갱신 ────────┤                                      │
```

#### 3.2 Chrome Runtime Message 스펙 (내부 통신으로 대체)

* **글로벌 오케스트레이션 개시 이벤트 (`INITIALIZE_SESSION`)**
```javascript
// From UI Dashboard to Background Service Worker
chrome.runtime.sendMessage({
  action: "INITIALIZE_SESSION",
  payload: {
    sessionId: "deeps-20260526",
    userPrompt: "Vue3 + Spring Boot 파일 업로드 컴포넌트 설계 요구사항",
    agents: ["chatgpt", "claude", "gemini"],
    options: { maxRounds: 3, timeoutMs: 15000 }
  }
});
```

  const responses = await getResponsesByRound(sessionId, currentRound);

  const round2Prompt = `[시스템 가이드: 동료 에이전트들의 교차 검증 라운드입니다]\n\n다음은 다른 AI 전문가들의 초안입니다. 본인의 의견과 비교하여 취약점을 날카롭게 지적하고 판단 근거를 요약하세요.\n\n${cleanContext}`;

  // 각 탭의 Content Script로 전송
  broadcastToAgents(sessionId, round2Prompt, currentRound + 1);
}

## 🛠️ 반성 및 아키텍처 재조정 (Refining the Gaps)

Claude가 지적한 '질문 100개 나올 만한 구멍'들을 메우기 위해, 실제 현업 SI 프로젝트를 리딩하시는 PL 관점에서 가장 가려우셨을 **핵심 기술적 디테일 3가지**를 먼저 고백하고 보완하겠습니다.

1. **인젝션 및 DOM 파싱 스펙 누락 (어떻게 긁어오고 넣을 것인가?)**
   * 각 LLM의 웹 페이지(ChatGPT, Claude 등)는 SPA(Single Page Application) 구조이며, 입력창과 전송 버튼의 CSS Class/ID가 수시로 바뀝니다. 
   * 이를 해결하기 위해 단순 Selector가 아니라 **`MutationObserver` API를 활용하여 텍스트 스트리밍 완료 시점(전송 버튼이 다시 활성화되는 시점 등)을 인터셉트하는 구체적인 핸들러 로직**이 설계서에 포함되어야 개발자가 코딩을 시작할 수 있습니다.
2. **세션 오케스트레이션의 물리적 한계 (진짜 New Thread가 가능한가?)**
   * 새 라운드마다 URL을 리다이렉트하면 페이지 로딩 시간(2~3초) 때문에 동기식 배리어의 병목이 심해집니다.
   * 현실적인 대안으로 **"동일 세션 안에서 진행하되, [토론 라운드 시작]이라는 명확한 Delimiter(구분자) 프롬프트를 주입하고, 시스템 백그라운드에서 이전 세션 메모리를 배열로 관리하는 방식"**이 훨씬 실용적입니다.
3. **크로스 도메인 보안 정책 (CORS 및 CSP 우회)**
   * 크롬 확장 프로그램의 `Manifest v3` 환경에서는 보안 정책(Content Security Policy)이 매우 엄격합니다. 대시보드 웹 페이지와 개별 LLM 탭 간의 대용량 컨텍스트 데이터 교환은 반드시 **`chrome.runtime.connect`를 통한 롱포링(Long-lived Connection) 양방향 채널**로 바인딩해야 데이터가 유실되지 않습니다.

## 🚀 Next Action: "더 이상의 질문이 없는 설계서"로 리팩토링

Claude의 날카로운 분석대로, 개발자가 질문 없이 코딩에 들어가려면 **Chrome Extension(Manifest v3) 기반의 실제 아키텍처**가 반영되어야 합니다. Deeps님이 최종 결정을 내리실 수 있도록 다음 장(Chapter)에서 다룰 **실무 명세서의 핵심 포함 사항**을 정리했습니다.

### 1. Chrome Extension 백그라운드 아키텍처 명세
* 웹 페이지 세션을 공유하기 위한 `Manifest v3` 환경 설정 (`permissions`, `host_permissions` 보안 우회 기법).

### 2. DOM 파싱 및 스트리밍 상태 머신(State Machine) 설계
* 각 LLM의 전송 버튼 및 답변 영역의 셀렉터 변경 리스크를 방지하기 위한 `MutationObserver` 핸들러 명세.
* AI가 답변을 작성 중인 상태(`STREAMING`)와 완료된 상태(`RESPONSE_END`)를 이벤트 리스너로 캐치하여 동기식 배리어(Lock)를 제어하는 상태 추적 메커니즘.

### 3. 라운드별 컨텍스트 격리(Delimiting) 프롬프트 명세
* 새 탭을 열지 않고 한 채팅방 내에서 컨텍스트 오염 없이 3라운드를 진행하기 위한 **`[SYSTEM BARRIER]` 격리 구분자 마스킹 가이드**.

## 4. 핵심 컴포넌트 소스코드 레벨 명세 (Core Code Specification)

### 4.1 메인 오케스트레이터 엔진 (`background/service_worker.js`)
라운드 상태 머신(State Machine)과 동기식 배리어(Lock), 서킷 브레이커(타임아웃 탈출 리스크 구조)를 제어하는 커널 코드입니다.

```javascript
// 에이전트 세션 및 라운드 상태 관리 백엔드 객체
const sessionStore = {
  activeSessionId: null,
  currentRound: 0,
  agents: {}, // { 'chatgpt': { port: null, r1_text: '', status: 'READY' } }
  config: { maxRounds: 3, timeoutMs: 15000 }
};

// 1. 대시보드 및 콘텐트 스크립트 연결 포트 리스너 (Long-lived Connection)
chrome.runtime.onConnect.addListener((port) => {
  if (port.name.startsWith("LLM_TAB_")) {
    const agentId = port.name.replace("LLM_TAB_", "");
    sessionStore.agents[agentId] = { port: port, responses: {}, status: "CONNECTED" };
    
    port.onMessage.addListener((msg) => {
      handleAgentMessage(agentId, msg);
    });
  }
});

// 2. 동기식 배리어(Lock) 제어 및 다음 라운드 트리거 로직
function verifyBarrierAndProceed() {
  const activeAgentIds = Object.keys(sessionStore.agents);
  const currentRound = sessionStore.currentRound;
  
  // 모든 에이전트가 COMPLETED 또는 TIMEOUT 상태인지 확인 (동기식 배리어)
  const isAllCleared = activeAgentIds.every(id => 
    sessionStore.agents[id].status === "COMPLETED" || sessionStore.agents[id].status === "TIMEOUT"
  );

  if (isAllCleared) {
    if (currentRound === 1) {
      triggerRoundTwo();
    } else if (currentRound === 2) {
      notifyDashboardRoundTwoComplete();
    }
  }
}

// 3. 서킷 브레이커 (Timeout Drop-out Handler)
function startRoundTimeoutTimer() {
  setTimeout(() => {
    Object.keys(sessionStore.agents).forEach(id => {
      if (sessionStore.agents[id].status === "STREAMING" || sessionStore.agents[id].status === "CONNECTED") {
        sessionStore.agents[id].status = "TIMEOUT";
        console.warn(`[System] Agent [${id}] Timeout Dropped.`);
      }
    });
    verifyBarrierAndProceed();
  }, sessionStore.config.timeoutMs);
}

// 4. 데이터 다이어트 및 정제 엔진 (컨텍스트 폭발 방지 정규식 구동)
function cleanContextText(rawText) {
  return rawText
    .replace(/(안녕하세요|반갑습니다|요약해 드리겠습니다|도움이 되셨길 바랍니다)/g, "") // 미사여구 제거
    .replace(/[\r\n]+/g, "\n") // 불필요한 개행 압축
    .trim();
}
```

### 4.2 DOM 파싱 및 스트리밍 감지 인젝터 (`content_scripts/chatgpt_injector.js` 예시)
SPA 구조에서 UI 변경 리스크를 최소화하고, 스트리밍 상태를 정밀하게 추적하기 위해 `MutationObserver`를 활용한 기법입니다. (Claude, Gemini 등 타 인젝터는 Selector 매핑 구조만 변경하여 동일 아키텍처로 구현)

```javascript
// 1. Service Worker 양방향 채널 커넥션 연결
const port = chrome.runtime.connect({ name: "LLM_TAB_CHATGPT" });

// 각 서비스별 DOM 셀렉터 명세 매핑 테이블
const DOM_SELECTORS = {
  inputArea: "div#prompt-textarea",
  sendButton: "button[data-testid='send-button']",
  lastResponseBlock: "div.agent-turn:last-child div.markdown"
};

// 2. 백엔드(Service Worker) 명령 수신 리스너
port.onMessage.addListener((msg) => {
  if (msg.action === "INJECT_PROMPT") {
    executeInject(msg.prompt);
  }
});

// 3. DOM 주입 및 가상 이벤트 트리거 (Anti-Bot Bypass)
function executeInject(promptText) {
  const inputEl = document.querySelector(DOM_SELECTORS.inputArea);
  if (!inputEl) return;

  inputEl.focus();
  // 가상 인풋 이벤트를 주입하여 프런트엔드 프레임워크(React/Vue) 상태 동기화
  inputEl.innerText = promptText;
  const event = new Event('input', { bubbles: true });
  inputEl.dispatchEvent(event);

  // 휴먼 딜레이 인젝션 (0.5초 대기 후 전송 버튼 클릭)
  setTimeout(() => {
    const sendBtn = document.querySelector(DOM_SELECTORS.sendButton);
    if (sendBtn && !sendBtn.disabled) {
      sendBtn.click();
      monitorStreamingStarted();
    }
  }, 500);
}

// 4. MutationObserver 기반 스트리밍 완결성 추적 알고리즘
function monitorStreamingStarted() {
  port.postMessage({ status: "STREAMING" });

  const observer = new MutationObserver((mutations, obs) => {
    const sendBtn = document.querySelector(DOM_SELECTORS.sendButton);
    // ChatGPT 특성: 생성 중일 때는 전송 버튼이 '중지(Stop)' 아이콘으로 바뀌거나 비활성화됨
    // 다시 전송 버튼이 활성화되는 시점이 스트리밍의 최종 종료 시점(RESPONSE_END)
    if (sendBtn && !sendBtn.disabled) {
      obs.disconnect(); // 옵저버 해제
      const finalOutput = document.querySelector(DOM_SELECTORS.lastResponseBlock).innerText;
      
      port.postMessage({
        status: "COMPLETED",
        text: finalOutput
      });
    }
  });

  observer.observe(document.body, { childList: true, subtree: true });
}
```

* **Round 2 (교차 토론 주입 프롬프트)**
  > `[SYSTEM SYSTEM_LOCK: ROUND_2_DEBATE]`
  > `[격리 구분자: 타 에이전트 의견 리포트]`
  > `${cleanContext}`
  > 
  > 위 내용은 Round 1에서 다른 AI 전문가들이 제안한 아키텍처 초안입니다. 본 대화방에서 본인이 직전에 했던 답변과 위 타인의 답변을 정밀 분석하여, **1) 타인 아키텍처의 치명적인 허점이나 누락 항목 1가지**, **2) 그럼에도 불구하고 내 아키텍처가 채택되어야 하는 명확한 기술적 판단 근거**를 계량화하여 논리적으로 반박하십시오.

# [Module 01] State Machine & Protocol Contract Specification

## 1. 글로벌 상태 머신 명세 (State Machine Spec)

시스템의 모든 라이프사이클은 아래의 결정론적 상태 트리(Deterministic State Tree)에 의해서만 트랜지션(Transition)되며, 예외 상황 발생 시 정의된 Abort/Rollback 정책을 엄격히 따릅니다.

### 1.1 State 레벨 정의 및 메모리 스토리지 스키마

```
                                  ┌──────────────┐
                                  │     IDLE     │
                                  └──────┬───────┘
                                         │ UI_START_SIGNAL
                                         ▼
                                  ┌──────────────┐
                                  │    QUEUED    │
                                  └──────┬───────┘
                                         │ DEPLOY_WORKER
                                         ▼
                                  ┌──────────────┐
                                  │ ROUND_RUNNING│◀──────────────────────┐
                                  └──────┬───────┘                       │
                                         │ ENGINE_STREAM_START           │
                                         ▼                               │
                                  ┌──────────────┐                       │
                                  │ BARRIER_WAIT │                       │ NEXT_ROUND_TRIGGER
                                  └──────┬───────┘                       │ (Round < MaxRound)
                    TIMEOUT_EXPIRED      │                               │
             ┌───────────────────────────┼───────────────────────────┐   │
             ▼                           ▼                           ▼   │
     ┌──────────────┐            ┌──────────────┐            ┌──────────────┐│
     │ TIMEOUT_DROP │            │  HITL_WAIT   │            │AGENT_CRITICAL││
     └──────┬───────┘            └──────┬───────┘            └──────┬───────┘│
            │                           │ UI_COMMIT_SIGNAL          │       │
            └───────────────────────────┼───────────────────────────┘       │
                                        ▼                                   │
                                 ┌───────────────┐                          │
                                 │FINAL_SYNTHESIS│──────────────────────────┘
                                 └──────┬────────┘
                                        │ SYNTHESIS_COMPLETE
                                        ▼
                                 ┌───────────────┐
                                 │   COMPLETED   │
                                 └───────────────┘
```

### 1.2 State Transition Matrix Table

| Current State | Event / Trigger | Next State | Condition / Side Effect |
| :--- | :--- | :--- | :--- |
| **IDLE** | `UI_START_SIGNAL` | **QUEUED** | 대시보드 입력 수신. 글로벌 세션 ID(UUID v4) 생성 및 메모리 적재. |
| **QUEUED** | `DEPLOY_WORKER` | **ROUND_RUNNING** | 대상 LLM 인젝터 탭 검증 완료. `currentRound = 1` 셋팅. |
| **ROUND_RUNNING** | `ENGINE_STREAM_START` | **BARRIER_WAIT** | 최소 1개 이상의 에이전트 포트에서 스트리밍 델타 수신 개시. 글로벌 타이머 스타트. |
| **BARRIER_WAIT** | `ALL_AGENTS_COMPLETE` | **HITL_WAIT** | 활성화된 모든 에이전트 상태가 `COMPLETED`로 변경됨. 타이머 Clear. |
| **BARRIER_WAIT** | `TIMEOUT_EXPIRED` | **TIMEOUT_DROP** | 설정된 `timeoutMs` 도달. 미응답 에이전트 포트 강제 `DROP` 및 더미 컨텍스트 삽입 후 `HITL_WAIT`로 강제 트랜지션. |
| **HITL_WAIT** | `UI_DEEP_DIVE_SIGNAL` | **HITL_WAIT** | 글로벌 배리어 유지 상태에서 특정 포트와 1:1 독립 롱포링 채널 격리 바인딩 및 샌드박스 UI 활성화. |
| **HITL_WAIT** | `UI_COMMIT_SIGNAL` | **FINAL_SYNTHESIS** | 사용자가 최종 채택안 체크 후 컴포즈 명령 전송. `currentRound` 값 검증. |
| **FINAL_SYNTHESIS**| `ROUND_LOOP_CONTINUE` | **ROUND_RUNNING**| `currentRound < maxRounds` 일 경우, 병합된 컨텍스트 프롬프트를 생성하여 `currentRound++` 후 재진입. |
| **FINAL_SYNTHESIS**| `SYNTHESIS_COMPLETE`  | **COMPLETED** | `currentRound == maxRounds` 조건 충족 시 최종 산출물 렌더링 후 세션 자원 풀 해제 및 파일 가동 종료. |
| **ANY_STATE** | `CRITICAL_EXCEPTION`  | **FAILED** | 브라우저 탭 크래시 혹은 DOM 셀렉터 파싱 불가 판정 시 에러 로그 구조화 저장 후 안전 비상 정지. |

# [System Specification] Multi-LLM Ensemble & Debate Platform
## 최종 요구사항 정의서 (Requirements Specification)

## 1. 프로젝트 및 제품 개요 (Product Spec)

### 1.1 제품 정의 (Product Definition)
본 시스템은 사용자의 크롬 브라우저에 로그인된 4대 LLM 서비스(ChatGPT, Claude, Gemini, Perplexity)의 액티브 세션을 공유하여 독립적인 API 비용 없이 **‘1단계: 발산(초안) $\rightarrow$ 2단계: 토론(교차 검증/근거) $\rightarrow$ 3단계: HITL형 수렴(사용자 판단 후 최종 빌드)’** 프로세스를 동기식 배리어(Synchronous Barrier)로 제어하는 크롬 확장 프로그램(Chrome Extension Manifest v3) 기반의 분산 오케스트레이션 플랫폼입니다.

### 1.2 대상 사용자 및 시나리오 (Target User & Scenario)
* **대상 사용자:** 복잡한 아키텍처 설계, 코드 정밀 검수, 다각도 리스크 검토가 필요한 IT 프로젝트 매니저(PM), 기술 리드(PL), 및 시니어 소프트웨어 엔지니어.
* **핵심 워크플로우:** 사용자 쿼리 입력 $\rightarrow$ 4대 LLM 동시 주입 및 1라운드 출력 $\rightarrow$ 출력물 가공 후 2라운드 상호 비판 프롬프트 자동 주입 $\rightarrow$ 2라운드 완료 후 사용자의 근거 심문(Deep Dive) 및 최종안 채택 $\rightarrow$ 최종 3라운드 정제 통합본 출력 및 종료.

### 1.3 시스템 범위 관리 (Scope Management)
* **In-Scope (포함 범위):**
  * Chrome Manifest v3 기반 Background Service Worker 코어 엔진.
  * 4대 LLM(ChatGPT, Claude, Gemini, Perplexity) 웹 SPA 서비스에 대응하는 개별 Content Script 인젝터 및 DOM 옵저버.
  * 단일 화면에서 4대 LLM의 스트리밍 응답을 동시 모니터링하고 라운드를 제어하는 대시보드 UI(HITL 콘솔).
  * 2라운드 종료 후 특정 LLM과 1:1로 판정 근거를 심문할 수 있는 세션 격리(Deep Dive) 모달 루프.
* **Out-of-Scope (제외 범위):**
  * 각 LLM 제공사의 공식 API 유료 키(API Key) 연동 기능 (철저히 브라우저 세션 기반 쿠키/웹스토리지 인증만 활용).
  * 로컬하드웨어 빌드 또는 온프레미스 서버 단독 구동형 대형 언어 모델(Local LLM) 연동 인터페이스.
  * 사용자 브라우저 외부의 별도 중앙 집중식 백엔드 데이터베이스(RDB/NoSQL) 구축 (모든 데이터는 크롬 내부 로컬 스토리지에 격리).

## 2. 기능 요구사항 정의서 (Functional Requirements)

### 2.1 오케스트레이션 및 라운드 제어 (Core Engine)
* **REQ-FUN-101: 글로벌 세션 초기화 (Initialize Session)**
  * 사용자가 대시보드에서 전송 버튼을 누르는 즉시 시스템은 UUID v4 규격의 고유 `sessionId`를 생성하고 상태를 `QUEUED`로 변경해야 한다.
  * 사용자가 선택한 활성 에이전트 목록(`activeAgents`)과 글로벌 제어 플래그(`maxRounds=3`, `timeoutMs=15000`)를 메모리에 셋업해야 한다.
* **REQ-FUN-102: 동기식 배리어 구현 (Synchronous Barrier Lock)**
  * 라운드 가동 시 시스템은 상태를 `BARRIER_WAIT`로 변경하고, 활성화된 모든 에이전트 포트로부터 응답 완료 시그널(`status: "COMPLETED"`)이 수집될 때까지 다음 라운드 프롬프트 생성을 엄격히 차단(Lock)해야 한다.
* **REQ-FUN-103: 서킷 브레이커 및 타임아웃 드롭 (Circuit Breaker & Timeout Drop)**
  * `BARRIER_WAIT` 상태 진입과 동시에 타이머를 가동하며, 특정 에이전트가 `timeoutMs(15000ms)` 내에 완료 시그널을 주지 않을 경우 시스템은 해당 에이전트의 상태를 `TIMEOUT`으로 강제 전환(Drop)하고, 나머지 성공한 에이전트의 결과물만 결합하여 다음 단계로 강제 진입해야 한다.
* **REQ-FUN-104: 데이터 다이어트 및 정제 (Context Trimming)**
  * 다음 라운드로 컨텍스트를 넘길 때, 에이전트가 출력한 원문에서 인사말, 맺음말, 불필요한 개행문자를 정규식(`/(안녕하세요|반갑습니다|도움이 되셨길|요약해)/g`)으로 완전 제거 가공한 후 순수 마크다운 본문만 파싱하여 주입해야 한다.

### 2.2 인젝션 및 DOM 스트리밍 제어 (Content Scripts)
* **REQ-FUN-201: 비동기 가상 입력 및 우회 (Input Emulation & Bypass)**
  * Content Script는 주입된 프롬프트 텍스트를 각 LLM의 입력 엘리먼트(`textarea`)의 `innerText`/`value`에 삽입한 후, 프런트엔드 렌더링 프레임워크(React/Vue)의 상태 동기화를 위해 가상 `Input Event`를 강제 디스패치(`dispatchEvent`)해야 한다. Anti-Bot 탐지 우회를 위해 입력 후 500ms의 휴먼 딜레이를 두고 전송 버튼 클릭 이벤트를 트리거해야 한다.
* **REQ-FUN-202: MutationObserver 기반 상태 추적 (Streaming Detection)**
  * 에이전트별 전송 버튼 클릭 후, 시스템은 `MutationObserver`를 body 전체에 바인딩하여 전송 버튼이 '비활성화(또는 중지 아이콘)'에서 '다시 활성화' 상태로 복구되는 시점을 스트리밍 종료(`RESPONSE_END`)로 판정하고, 최종 텍스트 노드의 내부 문자열을 추출하여 Background Worker로 전송해야 한다.

### 2.3 인간 참여형 제어 인터페이스 (HITL UI/UX)
* **REQ-FUN-301: 동적 그리드 멀티 뷰 (Dynamic Grid Dashboard)**
  * 활성화된 에이전트 수($N$)에 따라 대시보드는 `1/N` 세로 분할 카드로 실시간 스트리밍 텍스트를 렌더링해야 하며, 각 카드 상단에 에이전트 상태 배지(`READY`, `STREAMING`, `COMPLETED`, `TIMEOUT`)를 실시간 마킹해야 한다.
* **REQ-FUN-302: 근거 심문 모달 루프 (Isolated Deep Dive Loop)**
  * 2라운드(토론) 완료 후, 사용자가 특정 에이전트 카드의 `[근거 심문하기 🔍]` 버튼을 누르면 시스템은 상태를 `HITL_WAIT`로 고정하고, 타 에이전트 포트 통신을 일시 차단(Lock)한 채 해당 에이전트와 대시보드 간의 1:1 전용 독립 샌드박스 채팅 채널을 개설해야 한다.
* **REQ-FUN-303: 최종 결정 Commit 및 컴포즈 (Final Decision Commit)**
  * 사용자가 대시보드 화면에서 최종 채택할 요소(예: 에이전트 A의 코드 + 에이전트 B의 아키텍처)의 체크박스를 마킹하고 `[최종 빌드] Commit` 명령을 내리기 전까지 3라운드(최종 수렴) 진입을 원천 차단해야 한다.

## 3. 비기능 요구사항 정의서 (Non-Functional Requirements)

### 3.1 성능 및 확장성 (Performance & Scalability)
* **REQ-NFN-401: 통신 지연 최소화 (IPC Latency)**
  * Extension 백그라운드와 대시보드, 콘텐트 스크립트 간의 데이터 통신은 단방향 일회성 메시징을 금지하며, `chrome.runtime.connect` 기반의 양방향 롱포링 채널을 유지하여 내부 IPC 레이턴시를 50ms 이하로 통제해야 한다.
* **REQ-NFN-402: 가용 세션 제약 (Session Isolation)**
  * 브라우저 탭 자원 고갈 및 메모리 릭(Memory Leak)을 방지하기 위해 단일 확장 프로그램 인스턴스 내에서 동시 가동 가능한 디베이트 세션은 최대 1개로 제한하며, 새로운 세션 시작 시 이전 세션의 메모리 스토리지 버퍼를 완전히 Flash-Clear 해야 한다.

### 3.2 보안 및 안정성 (Security & Anti-Blocking)
* **REQ-NFN-501: 외부 유출 차단 및 보안 경계 (Security Boundary)**
  * 본 시스템이 다루는 사용자의 프롬프트, 소스코드, AI들의 토론 결과물은 외부 제3자 서버로 전송할 수 없으며, 모든 오케스트레이션 제어 로직은 크롬 내부 메모리 샌드박스 및 로컬 스토리지(`chrome.storage.local`) 내에서만 완결 처리되어야 한다.
* **REQ-NFN-502: 호트픽 셀렉터 가동 정책 (Hot-fix Selector Policy)**
  * 각 LLM 제공사의 정기 UI 업데이트로 인한 CSS 셀렉터 변경 리스크에 대응하기 위해, DOM 구조 셀렉터 명세는 하드코딩을 금지한다. 확장 프로그램 구동 시 로컬 스토리지에 캐싱된 셀렉터 스키마 JSON 데이터를 우선 참조하도록 구현해야 한다.

## 4. 기계적 상태 전이 스펙 (State Machine Spec)

구현자가 자의적으로 예외를 처리하지 않도록 시스템 상태 코드 및 전이 조건(Transition Rules)을 기계적 단위로 명시합니다.

```
[IDLE] ──(UI_START_SIGNAL)──> [QUEUED] ──(DEPLOY_WORKER)──> [ROUND_RUNNING]
                                                                  │
                                                        (ENGINE_STREAM_START)
                                                                  │
                                                                  ▼
[HITL_WAIT] <──(ALL_AGENTS_COMPLETE / TIMEOUT_EXPIRED)── [BARRIER_WAIT]
     │
(UI_COMMIT_SIGNAL)
     │
     ▼
[FINAL_SYNTHESIS] ──(ROUND_LOOP_CONTINUE: Round < Max)──> [ROUND_RUNNING]
     │
(SYNTHESIS_COMPLETE: Round == Max)
     │
     ▼
[COMPLETED]
```

### 4.1 상태 전이 규칙 제약서
1. `BARRIER_WAIT` 상태에서 타이머가 `15000ms`를 경과하는 순간, `ALL_AGENTS_COMPLETE` 이벤트의 충족 여부와 무관하게 시스템 강제 인터럽트가 발생하며 상태는 즉시 `TIMEOUT_DROP`을 경유하여 `HITL_WAIT`로 강제 강하한다.
2. 어떠한 라운드에서도 `CRITICAL_EXCEPTION`(탭 종료, DOM 탐색 파괴)이 감지되면 오케스트레이터는 즉시 실행 스레드를 중단하고 상태를 `FAILED`로 변경한 뒤, 로컬 스토리지에 에러 로그 패킷(`ERR_CODE`)을 적재하고 대시보드에 비상 정지 UI를 출력해야 한다.

추가된 요구사항이 기술적으로 어떻게 구현 가이드라인으로 녹아드는지 매핑해 드립니다.

## 🛠️ 추가 요구사항에 대한 기술적 검토 및 구현 매핑

### 1. 웹브라우저 로그인 계정 기반 동적 탐색 (Dynamic LLM Detection)
* **어떻게 구현하는가:** 확장 프로그램이 켜지면 Background Service Worker가 각 LLM 서비스의 특정 endpoint(예: ChatGPT의 `/api/auth/session` 또는 Claude의 `/api/organizations`)에 백그라운드 페치(Fetch)를 날리거나, `chrome.cookies.get`을 통해 인증 토큰 존재 여부를 확인합니다.
* **결정론적 규칙 (`No Guess`):** * 성공(`HTTP 200` 또는 토큰 존재) ➡️ 대시보드 UI에 **'활성화(Active)'** 상태로 표시.
  * 실패(`HTTP 401/403` 또는 토큰 없음) ➡️ 대시보드 UI에 **'로그인 필요(Unauthorized)'** 경고 및 링크 노출.

### 2. 선택 또는 전체 협업 (Dynamic Target Filtering)
* **어떻게 구현하는가:** 대시보드 상단에 체크박스 또는 토글 리스트를 배치합니다. 글로벌 전송 버튼을 누를 때, 전체가 아닌 **'선택되어 활성화된 에이전트 리스트(Array)'**만 매개변수로 상태 머신에 전달합니다.
* **결정론적 규칙 (`No Guess`):**
  * 2라운드 동기식 배리어(Lock)는 모든 모델이 아니라, **사용자가 선택한 에이전트들의 포트 응답만 수집 완료되면 즉시 해제**되도록 동적 카운팅 로직을 적용합니다.

# [Milestone 01] 기능 및 비기능 요구사항정의서
**Project: Multi-LLM Ensemble & Debate Extension (v1.0)**

## 1. 일반 개요 (General Overview)
* **시스템 형태:** Chrome Extension Manifest v3 단독 아키텍처
* **프런트엔드 스택:** Vue3 (Vite 빌드 기반 SFC 구조)
* **로컬 데이터 저장소:** IndexedDB (세션, 로그, 셀렉터 데이터 저장용) & chrome.storage.local (단순 플래그성 설정용)
* **대상 언어:** 한국어 기본 지원 (i18n 확장 가능 구조 고려)

## 2. 기능 요구사항 (Functional Requirements)

### 2.1 세션 동적 탐색 및 에이전트 필터링 (New)
* **REQ-FUN-101: 브라우저 로그인 세션 동적 탐색**
  * **설명:** 확장 프로그램 가동 즉시 사용자의 크롬 브라우저 쿠키 및 인증 세션을 조회하여 접속 가능한 LLM 목록을 동적으로 식별해야 한다.
  * **상세 규칙:** 4대 LLM(ChatGPT, Claude, Gemini, Perplexity)의 특정 인증 Endpoint에 백그라운드 요청을 송신하여 `HTTP 200` 회신 시 'ACTIVE(활성화)', 그 외의 경우 'LOGIN_REQUIRED(로그인 필요)' 상태로 대시보드 UI에 매핑해야 한다.
* **REQ-FUN-102: 대상 에이전트 동적 선택 및 필터링**
  * **설명:** 사용자는 대시보드 상단에 노출된 에이전트 활성 목록에서 '전체 선택' 또는 '개별 체크박스 선택'을 통해 협업에 참여할 대상을 동적으로 지정할 수 있어야 한다.
  * **상세 규칙:** 선택되지 않은 에이전트는 해당 디베이트 세션의 오케스트레이션 대상 스레드에서 즉시 제외되며, 라운드 동기화 배리어 카운트에서도 배제되어야 한다.

### 2.2 오케스트레이션 및 라운드 제어
* **REQ-FUN-201: 글로벌 세션 초기화**
  * **설명:** 사용자가 쿼리 입력 후 전송 시, UUID v4 포맷의 `sessionId`를 생성하고 상태를 `QUEUED`로 전환해야 한다.
* **REQ-FUN-202: 동기식 배리어 구현 (Synchronous Barrier Lock)**
  * **설명:** 각 라운드 구동 시 시스템은 `BARRIER_WAIT` 상태로 진입하며, **REQ-FUN-102**에서 사용자가 선택한 모든 에이전트로부터 응답 완료 시그널(`status: "COMPLETED"`)이 수집될 때까지 다음 라운드 진행을 엄격히 차단해야 한다.
* **REQ-FUN-203: 서킷 브레이커 및 타임아웃 강제 강하**
  * **설명:** `BARRIER_WAIT` 상태 진입 후 `15000ms`를 경과하면 타이머 인터럽트가 발생하여 미응답 에이전트를 강제 `TIMEOUT` 처리하고, 성공한 에이전트의 결과물만 취합하여 다음 단계로 강제 진입해야 한다.
* **REQ-FUN-204: 컨텍스트 다이어트 (Trimming)**
  * **설명:** 다음 라운드로 텍스트를 인젝션하기 전, 에이전트 출력물에서 불필요한 미사여구(인사말, 맺음말) 및 중복 개행을 정규식을 통해 완전 제거 정제해야 한다.

### 2.3 인간 참여형 제어 인터페이스 (HITL UI/UX)
* **REQ-FUN-301: Vue3 기반 동적 그리드 대시보드**
  * **설명:** 사용자가 선택한 에이전트 수($N$)에 따라 화면을 `1/N` 세로 분할 카드로 가변 렌더링하고, 스트리밍 상태 배지를 실시간 마킹해야 한다.
* **REQ-FUN-302: 근거 심문 격리 루프 (Isolated Deep Dive)**
  * **설명:** 2라운드(토론) 종료 후 사용자가 특정 카드 하단의 `[근거 심문하기 🔍]` 버튼을 누르면 시스템은 `HITL_WAIT` 상태로 진입하고, 해당 에이전트의 포트와 대시보드 간 1:1 독립 샌드박스 채팅 세션을 활성화해야 한다.
* **REQ-FUN-303: 최종 결정 Commit 제어**
  * **설명:** 사용자가 대시보드에서 최종 채택할 항목을 마킹하고 `[최종 빌드] Commit` 버튼을 누르기 전까지 최종 3라운드(수렴) 진입을 원천 차단해야 한다.

## 3. 비기능 요구사항 (Non-Functional Requirements)

### 3.1 성능 및 안정성
* **REQ-NFN-401: 장지속 채널 통신 (IPC Latency)**
  * **설명:** Background Service Worker와 각 LLM 탭(Content Script) 간 통신은 `chrome.runtime.connect` 기반의 롱포링 채널을 유지하여 데이터 누실을 방지하고 내부 레이턴시를 50ms 이하로 통제해야 한다.

  * **설명:** LLM 서비스의 DOM 구조 변경 리스크에 대응하기 위해 모든 CSS 셀렉터는 원격 서버 또는 스토리지 JSON 구조를 동적 참조하도록 설계하며, 코드 내 하드코딩을 전면 금지한다.
* **REQ-NFN-403: 로컬 보안 경계 (Security Boundary)**
  * **설명:** 시스템 내부에서 발생하는 모든 쿼리, 소스코드, 토론 로그는 외부 독립 서버로 유출되거나 전송될 수 없으며, 크롬 내부 메모리 샌드박스 및 IndexedDB 내부에서만 완결 저장되어야 한다.

## 📋 핵심 요구사항 및 작동 메커니즘 개요

### 1. 진입 단계: 브라우저 세션 기반의 동적 환경 탐색
* 별도의 유료 API Key 연동 없이, 사용자가 현재 크롬 브라우저에서 로그인하여 사용 중인 **ChatGPT, Claude, Gemini, Perplexity** 등의 액티브 세션을 시스템이 동적으로 자동 감지합니다.
* 사용자는 감지된 AI 목록 중 이번 프로젝트 협업에 참여시킬 AI들을 전체 또는 개별적으로 선택하여 조합할 수 있습니다.

### 2. 구동 단계: 3라운드 제약 기반의 동기식 오케스트레이션
AI들이 자기 스타일대로 횡설수설하거나 컨텍스트가 폭발하지 않도록, 시스템이 중심에서 딱 **3번의 라운드 프로토콜**을 기계적으로 강제합니다.

* **1라운드 (독립적 발산):** 사용자의 요구사항을 받은 AI들이 서로의 답변을 보지 못한 채, 각자의 성향(Persona)에 맞춰 독자적인 초안을 동시에 도출합니다.
* **동기식 배리어 (Synchronous Barrier):** 선택된 모든 AI의 응답이 완료되거나 타임아웃(15초)이 되기 전까지 시스템은 절대 다음 단계로 넘어가지 않고 브레이크를 겁니다.
* **2라운드 (교차 토론 및 근거 제시):** 1라운드 답변들을 정제하여 서로에게 교차 주입합니다. 각 AI는 **"상대방 방안의 치명적인 허점"**과 **"내 방안이 채택되어야 하는 명확한 기술적 근거"**를 매섭게 상호 비판하며 리포트를 제출합니다.

### 3. 제어 및 결론 단계: 인간 참여형(HITL) 루프와 최종 빌드
* **근거 심문 (Deep Dive):** 2라운드가 끝나면 사용자가 특정 AI를 콕 집어 1:1로 판정 근거를 다시 추적하고 심문할 수 있는 독립 세션 모달이 열립니다.
* **최종 수렴 (Commit):** AI들에게 최종 결정을 맡기지 않습니다. 사용자가 AI들의 상호 비판 근거를 바탕으로 장점만 골라 체크한 뒤 **최종 승인(Commit)**을 내리면, 시스템이 이를 종합하여 추가 질문이 필요 없는 완벽한 최종 아웃풋을 빌드하고 프로세스를 종료합니다.

## 📋 핵심 요구사항 및 작동 메커니즘 개요

### 1. 진입 단계: 브라우저 세션 기반의 동적 환경 탐색
* 별도의 유료 API Key 연동 없이, 사용자가 현재 크롬 브라우저에서 로그인하여 사용 중인 **ChatGPT, Claude, Gemini, Perplexity** 등의 액티브 세션을 시스템이 동적으로 자동 감지합니다.
* 사용자는 감지된 AI 목록 중 이번 프로젝트 협업에 참여시킬 AI들을 전체 또는 개별적으로 선택하여 조합할 수 있습니다.

### 2. 구동 단계: 3라운드 제약 기반의 동기식 오케스트레이션
AI들이 자기 스타일대로 횡설수설하거나 컨텍스트가 폭발하지 않도록, 시스템이 중심에서 딱 **3번의 라운드 프로토콜**을 기계적으로 강제합니다.

* **1라운드 (독립적 발산):** 사용자의 요구사항을 받은 AI들이 서로의 답변을 보지 못한 채, 각자의 성향(Persona)에 맞춰 독자적인 초안을 동시에 도출합니다.
* **동기식 배리어 (Synchronous Barrier):** 선택된 모든 AI의 응답이 완료되거나 타임아웃(15초)이 되기 전까지 시스템은 절대 다음 단계로 넘어가지 않고 브레이크를 겁니다.
* **2라운드 (교차 토론 및 근거 제시):** 1라운드 답변들을 정제하여 서로에게 교차 주입합니다. 각 AI는 **"상대방 방안의 치명적인 허점"**과 **"내 방안이 채택되어야 하는 명확한 기술적 근거"**를 매섭게 상호 비판하며 리포트를 제출합니다.

### 3. 제어 및 결론 단계: 인간 참여형(HITL) 루프와 최종 빌드
* **근거 심문 (Deep Dive):** 2라운드가 끝나면 사용자가 특정 AI를 콕 집어 1:1로 판정 근거를 다시 추적하고 심문할 수 있는 독립 세션 모달이 열립니다.
* **최종 수렴 (Commit):** AI들에게 최종 결정을 맡기지 않습니다. 사용자가 AI들의 상호 비판 근거를 바탕으로 장점만 골라 체크한 뒤 **최종 승인(Commit)**을 내리면, 시스템이 이를 종합하여 추가 질문이 필요 없는 완벽한 최종 아웃풋을 빌드하고 프로세스를 종료합니다.

## 🛠️ [Deeps Architecture v1.0] 1단계: 최종 요구사항정의서 (SRS)

자가 해석(`No Guess`)과 추후 결정(`TBD`)을 완전히 소멸시키고, 추가된 2가지 조건(동적 세션 탐색 및 필터링)을 정밀 반영한 **최종 완결형 요구사항정의서**입니다.

### 1. 제품 범위 및 제약 사항 (Scope & Constraints)
* **환경 (Environment):** Chrome Extension Manifest v3 단독 구동.
* **기술 스택 (Tech Stack):** Vue3 (대시보드 UI) + IndexedDB (로컬 세션 및 로그 저장).
* **접속 방식 (Auth):** 유료 API Key 없이, 현재 브라우저에 로그인된 LLM 웹 세션(Cookie/Auth Header)을 그대로 공유 및 가로채기(ChatHub 방식).

### 2. 단계별 기능 요구사항 명세 (Functional Requirements)

#### 2.1 [진입] 동적 환경 탐색 및 필터링
* **REQ-INIT-01 (세션 탐색):** 확장 프로그램 실행 시, 4대 LLM(ChatGPT, Claude, Gemini, Perplexity)의 인증 및 세션 Endpoint에 백그라운드 Fetch를 날려 사용자의 로그인 여부를 동적으로 체크하고 UI에 활성/비활성 상태를 마킹한다.
* **REQ-INIT-02 (에이전트 선택):** 사용자는 로그인된 LLM 목록 중 이번 협업에 참여할 대상을 최소 1개부터 전체까지 체크박스로 동적 선택한다. 선택되지 않은 모델은 오케스트레이션 루프 및 배리어 카운트에서 완전히 제외된다.

#### 2.2 [구동] 3라운드 동기식 오케스트레이션
* **REQ-EXEC-01 (1라운드: 독립 발산):** 사용자가 대시보드에 글로벌 프롬프트를 입력하면, 시스템은 선택된 LLM 탭(Content Script)에 입력을 주입하고 전송 버튼을 트리거한다. 이때 각 LLM은 서로의 답변을 보지 못하는 Blind 상태로 초안을 생성한다.
* **REQ-EXEC-02 (동기식 배리어 락):** 시스템은 `BARRIER_WAIT` 상태로 진입하며, 선택된 모든 LLM 포트로부터 완료 시그널(`COMPLETED`)이 수집될 때까지 다음 단계를 엄격히 블로킹한다. 단, 특정 모델이 `15000ms` 동안 응답이 없으면 서킷 브레이커가 발동하여 해당 모델을 강제 `TIMEOUT_DROP` 처리하고 진행한다.
* **REQ-EXEC-03 (컨텍스트 다이어트):** 수집된 답변들에서 인사말, 맺음말, 무의미한 개행을 정규식으로 정제한 후 순수 핵심 내용만 파싱한다.
* **REQ-EXEC-04 (2라운드: 교차 토론):** 정제된 타 모델의 답변을 격리 구분자(`Delimiter`)와 함께 각 LLM 탭에 다시 주입하여, "상대방 방안의 허점 지적" 및 "내 방안의 기술적 근거 제시"를 강제하는 프로그래밍된 프롬프트를 실행한다.

#### 2.3 [제어 및 결론] 인간 참여형(HITL) 루프
* **REQ-HITL-01 (근거 심문: Deep Dive):** 2라운드 종료 후, 사용자가 특정 AI 카드의 `[근거 심문하기]`를 누르면 글로벌 오케스트레이션을 일시 정지하고, 해당 LLM 탭과 대시보드 간의 1:1 독립 샌드박스 채팅 세션을 활성화하여 사용자가 직접 추가 질의를 할 수 있도록 제어권을 넘긴다.
* **REQ-HITL-02 (최종 Commit 및 수렴):** 사용자가 각 AI의 응답 카드에서 채택할 요소를 체크하고 `[최종 빌드] Commit` 버튼을 누르기 전까지 최종 3라운드 진입을 원천 차단하며, 사용자의 승인이 떨어지면 이를 종합한 최종 결과물을 빌드(3라운드)하고 세션을 종료한다.

### ⚠️ Mistral 코드 검수 및 방어벽 가이드

우리가 동결한 **요구사항정의서(SRS)**의 룰을 Mistral이 지켰는지 검수하기 위해, 딱 **3가지 핵심 구조**가 반영되었는지 코드에서 Ctrl+F로 바로 확인해 보세요.

1. **`chrome.runtime.connect` (롱포링 채널):** 단발성 `sendMessage`로 대충 짜놓았다면 대용량 스트리밍 데이터 유실됩니다. 무조건 장지속(Long-lived) 채널로 짜여 있는지 확인하십시오.
2. **`MutationObserver` (종료 감지):** 단순히 `setTimeout`이나 느슨한 타이머로 끝나는 시점을 예측하게 짰다면, LLM 답변이 길어질 때 배리어(Lock)가 깨집니다. DOM 변화를 추적하는지 확인하십시오.
3. **`Dynamic Target Filtering` (로그인 세션 배열):** 4대 모델 고정이 아니라, 사용자가 선택한 에이전트 리스트(`selectedAgents` 배열)만 루프를 돌며 배리어 카운트를 동적으로 계산하는지 확인하십시오.
