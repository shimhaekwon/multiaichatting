## 1. 항목별 검토

### C1 — STOP DONE 통보 ✅ (보강 1건)

| 파일 | Plan 내용 | 판정 |
|------|-----------|------|
| `content-script/dom-adapter.ts` (주) | `stopObserve`를 onConnect 콜백 스코프로 hoist → SEND_PROMPT에서 외부 변수 할당 → STOP에서 `stopObserve?.()` + `port.postMessage({action:'DONE'})`, `onDisconnect`에서 정리 | ✅ 정확(원검토 방향과 동일) |
| `app/bots/dom-adapter-bot.ts` (보강) | abort catch에서 port-dead 시 `params.onEvent({type:'DONE'})` + `cleanup()` | ✅ 타당 |

- **double-DONE 분석(Plan 주장)**: "CS가 DONE 보내면 `port.onMessage`가 먼저 처리 → `controller.close()`. catch는 `postMessage` 실패 시에만 진입" → **맞음**(abort 경로 한정).

🟡 **보강 — CS 내부 double-DONE 경합 (저위험)**
- Plan의 double-DONE 분석은 *abort 경로(파일2)*만 커버. **파일1에서 자연 done(observeResponse가 done 판정→DONE)과 STOP이 거의 동시에 발생**하면 STOP 핸들러가 DONE을 한 번 더 보냄.
- 보통은 첫 DONE에서 app이 `cleanup()`(port.disconnect)하여 두 번째 DONE이 app에 도달 못 해 무해. 단 동일 tick이면 background 릴레이 타이밍상 `controller.close()` 재호출 → 콘솔 에러 가능.
- **권장**: STOP에서 done 가드. observeResponse done 시 외부에 신호를 주고 `if (stopObserve) { stopObserve(); stopObserve = null; port.postMessage({action:'DONE'}) }`로 묶기. (현재 Plan은 무조건 postMessage)

### M1 — connectToCSWithRetry 직렬화 ✅ (완전)

- Plan: `return new Promise(...)` → `const result = await new Promise(...)` + `if (result !== null) return result`, 실패 시 루프 계속.
- **판정: 정확. 원지적 완전 해결.** 재시도가 실제로 동작함.
- 참고: 100ms 판정 + 300ms delay × 최대 3회 ≈ 1.2초. 매우 느린 CS 주입은 여전히 실패 가능하나 best-effort로 충분.

### M2 — proxy-fetch.ts 삭제 ✅

- Plan: 파일 삭제, import 0건(grep 확인 완료).
- **판정: 타당.** 빌드 검증으로 확정.

## 2. 검증 절차 보강

Plan의 검증(① `tsc --noEmit` ② `yarn build` ③ grep proxyFetch ④ deepseek STOP E2E ⑤ 미로그인 ERR_002)은 적절. 아래 2건 추가 권장:

- **해피패스 회귀**: STOP 없는 정상 3라운드(IT-01)가 C1 변경(`onDisconnect`의 `stopObserve` null화)으로 안 깨졌는지 — 자연 done 경로 확인.
- 🟠 **인접 hang 경로 (C1과 별개, 확인 필요)**: `background/index.ts`의 `cleanup`은 `csPort.onDisconnect` 시 `csPort.disconnect()`만 하고 **`appPort`에 DONE/ERROR를 통보하지 않음.** 즉 응답 대기 중 **CS 탭이 닫히거나 CS가 죽으면** app의 `doSendMessage`가 종료 신호를 못 받아 hang 잔존. dom-adapter-bot의 fallback은 *app↔bg* port 기준이라 *bg↔CS* 단절은 미커버. → `cleanup`에서 `appPort.postMessage({action:'ERROR', code:'ERR_002'})` 류 종료 통보 추가 검토 권장.
