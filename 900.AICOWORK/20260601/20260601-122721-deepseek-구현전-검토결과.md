## 2. C1 상세 — STOP DONE 미통보 → 라운드 hang (CRITICAL)

### 2.1 문제 재현

**deepseek(stopButton='') 기준 흐름**:

```
Round 전송 → Promise.all(slots.map(async (s) => { ... await s.chat.sendMessage(full) }))
                                               ↓
                                    DomAdapterBot.doSendMessage()
                                     port → BG relay → CS
                                               ↓
                                    observeResponse timer 300ms polling
                                               ↓
User STOP → useRoundEngine.stop() → slots.forEach(s.chat.stopGenerating)
                                               ↓
                              chatState.abortController.abort()
                                               ↓
                            DomAdapterBot: port.postMessage({action:'STOP'})
                                               ↓
                            BG relay → CS port.onMessage
                                               ↓
                            CS: findStop()?.click()  // deepseek: stopButton='' → null → click無
                                               ↓
                            observeResponse timer 계속 실행
                            deepseek 생성 지속 → lastChange 계속 갱신
                            done = !!lastText && Date.now()-lastChange > 1500ms → 영원히 false
                                               ↓
                            CS → DONE 안 보냄 → for await 미종료
                                               ↓
                            Promise.all hang → history.push / persist 미실행
```

### 2.2 문제 위치 (코드 라인 확정)

**`content-script/dom-adapter.ts:91-119`**:

```typescript
chrome.runtime.onConnect.addListener((port) => {
  let currentSelectors: AdapterSelectors | undefined;

  port.onMessage.addListener(async (msg: any) => {
    if (msg.action === "SEND_PROMPT") {
      // ... 생략 ...
      const stopObserve = observeResponse(s, (text, done) => {
        // ← SEND_PROMPT scope
        port.postMessage(
          done
            ? { action: "DONE" }
            : { action: "RESPONSE_CHUNK", content: text },
        );
      });
      port.onDisconnect.addListener(() => stopObserve());
    } else if (msg.action === "STOP") {
      if (currentSelectors) {
        findStop(currentSelectors)?.click(); // ← DONE 통보·타이머 중단 누락
      }
    }
  });
});
```

**`app/bots/dom-adapter-bot.ts:25-31`**:

```typescript
params.signal?.addEventListener("abort", () => {
  try {
    port.postMessage({ action: "STOP" }); // ← DONE·cleanup 누락
  } catch {
    /* port may be gone */
  }
});
```

### 2.3 수정안

**File 1**: `content-script/dom-adapter.ts`

변경: `stopObserve`를 hoist하고 STOP 핸들러에서 타이머 중단 + DONE 통보

```typescript
chrome.runtime.onConnect.addListener((port) => {
  let currentSelectors: AdapterSelectors | undefined;
  let stopObserve: (() => void) | null = null; // ← hoist

  port.onMessage.addListener(async (msg: any) => {
    if (msg.action === "SEND_PROMPT") {
      // ... 생략 ...
      stopObserve = observeResponse(s, (text, done) => {
        port.postMessage(
          done
            ? { action: "DONE" }
            : { action: "RESPONSE_CHUNK", content: text },
        );
      });
      port.onDisconnect.addListener(() => {
        stopObserve?.();
        stopObserve = null;
      });
    } else if (msg.action === "STOP") {
      if (currentSelectors) findStop(currentSelectors)?.click();
      stopObserve?.(); // 타이머 즉시 중단
      stopObserve = null;
      port.postMessage({ action: "DONE" }); // 스트림 종료 통보
    }
  });
});
```

**File 2**: `app/bots/dom-adapter-bot.ts`

변경: abort 시 CS 무응답 fallback으로 DONE 직접 통보

```typescript
params.signal?.addEventListener("abort", () => {
  try {
    port.postMessage({ action: "STOP" });
  } catch {
    // CS가 응답할 수 없는 상태 → DONE 직접 close
    params.onEvent({ type: "DONE" });
    cleanup();
  }
});
```

> **double-DONE 안전성**: CS가 정상 DONE을 보내면 port.onMessage가 먼저 처리되어 controller.close().  
> catch 블록은 port.postMessage 실패(CS 사망) 시에만 진입 → 동시 DONE 가능성 없음.

## 3. M1 상세 — connectToCSWithRetry 재시도 무효 (MEDIUM)

### 3.1 문제 재현

**`background/index.ts:37-64`**:

```typescript
async function connectToCSWithRetry(tabId, name, retries = 3, delay = 300) {
  for (let i = 0; i < retries; i++) {
    try {
      const port = chrome.tabs.connect(tabId, { name });
      return new Promise((resolve) => {
        // ← 루프 첫 회에서 즉시 탈출
        const onDisconnect = () => {
          port.onDisconnect.removeListener(onDisconnect);
          resolve(null);
        };
        port.onDisconnect.addListener(onDisconnect);
        setTimeout(() => {
          port.onDisconnect.removeListener(onDisconnect);
          resolve(port);
        }, 100);
      });
    } catch {
      if (i < retries - 1) await new Promise((r) => setTimeout(r, delay));
    }
  }
  return null;
}
```

`return new Promise(...)`는 루프를 첫 iteration에서 즉시 탈출시킴.

- CS 미준비 시 onDisconnect가 ~0ms에 불 → `resolve(null)` → 함수 종료 (재시도 0회)
- `catch` 블록은 `chrome.tabs.connect` 자체 실패 시에만 도달 (tabId 유효 시 거의 발생 안 함)

### 3.2 수정안

```typescript
async function connectToCSWithRetry(
  tabId: number,
  name: string,
  retries = 3,
  delay = 300,
): Promise<chrome.runtime.Port | null> {
  for (let i = 0; i < retries; i++) {
    try {
      const port = chrome.tabs.connect(tabId, { name });
      const result = await new Promise<chrome.runtime.Port | null>(
        (resolve) => {
          port.onDisconnect.addListener(() => resolve(null));
          setTimeout(() => resolve(port), 100);
        },
      );
      if (result !== null) return result; // 성공 시 탈출
    } catch {
      /* tabs.connect 자체 실패 — 다음 iteration */
    }
    if (i < retries - 1) await new Promise((r) => setTimeout(r, delay));
  }
  return null;
}
```

핵심 차이: `return new Promise` → `await` + null 체크. 실패 시 루프 지속.

## 5. LOW 이슈 검토

### L1 — secretChatAtom atomWithImmer (명세 문구 차이)

`state/round.ts:15`: `export const secretChatAtom = atomWithImmer<SecretChat | null>(null)`

222는 `atom()` 명시, 현재는 `atomWithImmer`. 기능 차이 없음(둘 다 휘발성, 스토리지 미사용).
secretChat은 recipe 기반 업데이트(`d.messages.push(...)`)가 필요해 immer 사용이 적합.

**판정**: 명세 문구 불일치이나 기능상 올바름 → 유지

### L2 — generating useMemo 의존성

`use-round-engine.ts:59`: `const generating = useMemo(() => slots.some((s) => s.chat.generating), [slots])`

의존성 배열에 `s.chat.generating`이 직접 명시되지 않았으나,
`slots = useMemo(..., [selected, ...allChats])` → `allChats[i]` 참조 변경 → `slots` 재생성 → `generating` 재계산.

**판정**: 참조 체인으로 정상 동작하나, 정적 분석 도구 경고 가능. 필요 시 `slots` 대신 각 chat.generating을 개별 의존성으로 명시 가능.

### L3 — 자동 테스트 0

220 TC-01~14 존재하나 미자동화. 219 Phase 정책이 수동 검증 위주이므로 Phase 정책 범위 내.

**판정**: 필요 시 추후 도입. 현재는 유지.

### L4 — fillPrompt 300ms 고정 대기

`dom-adapter.ts:107`: `await new Promise((r) => setTimeout(r, 300))`

slow site에서 부족 가능. best-effort이며 사이트별 튜닝은 v1 범위 외.

**판정**: 유지.
