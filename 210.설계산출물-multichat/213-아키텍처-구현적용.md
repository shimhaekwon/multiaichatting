# 213. 아키텍처 구현적용

> **자기완결**. 핵심 결정 = **라운드 엔진은 앱(app.html 탭) 컨텍스트** + 매 라운드 `chrome.storage.session` 스냅샷(원안 "SW 라운드엔진"에서 변경). [대조용 초안: 200-초안/210·240]

## 1. 원안 아키텍처 → ChatHub 레이어 매핑

| 원안 레이어 | ChatHub 구현 |
|----------------|--------------|
| UI Layer | `app.html`→`src/app/main.tsx`→router `/`=`MultiBotChatPanel`(2×3 그리드, **메인 탭**) [N-1] |
| Background (Round Engine/Router/Storage/MD) | **분할**: Round Engine·Context·MD는 **앱 컨텍스트**(`use-round-engine.ts` 등), Background는 **DOM 봇 탭 메시지 라우팅 + 명령**만 담당 |
| Content Script (Adapter/DOM) | **6종 전부** → `content-script/dom-adapter.ts`(로그인 탭 DOM 조작). proxyFetch/inpage-proxy 폐기 [2026-05-29 all-DOM] |
| LLM Web Services (6종) | 각 사이트 탭(로그인 세션) |

## 2. 라운드 엔진 위치 결정 (중요)

**결정: 앱(app.html 탭) 컨텍스트에 둔다.**

- **근거**: 엔진을 앱(React)에 두면 `bot.sendMessage()`를 직접 `await`하며 슬롯별 `useChat` 스트리밍·상태를 그대로 쓸 수 있어 단순하다. 6종 DOM 봇은 `DomAdapterBot`이 background 릴레이를 통해 탭 content script와 통신하므로, 엔진은 봇 메커니즘을 모른 채 균일 호출만 한다 → SW로 옮길 이유 없음.
- **trade-off / 보완**: 앱 탭이 닫히면 진행이 멈출 수 있다. → **매 라운드 완료 시 `chrome.storage.session`에 스냅샷**(214 §5)을 저장하고, 재오픈 시 재개(best-effort). 원안의 SW 30초 수명 우려는 "엔진이 SW가 아니므로" 직접 해당되지 않으며, DOM 봇의 탭 작업은 background를 단순 중계로만 사용한다.
- **원안 부분수정 명시**: 원안은 "Background SW = Round Engine"이나, 본 구현은 **앱 컨텍스트 엔진 + background는 라우터**. (사용자 인지 완료)

## 3. 봇 메커니즘 — 전 봇 DOM 통일 (핵심) [2026-05-29]

모든 봇은 동일 인터페이스 `AbstractBot.sendMessage({prompt, signal}) → AsyncIterable<{text}>` 를 노출하고, 라운드 엔진은 **메커니즘을 모른 채** 균일 호출한다. **Phase 0 실증으로 내부 API(ChatGPT/Claude/Perplexity)가 전부 깨져 6종 모두 DOM 어댑터로 통일**(하이브리드 폐기).

### 단일 경로 — DOM 어댑터 (탭 경유, 6종 공통)

DOM은 해당 사이트 페이지에서만 가능하므로, **content script가 실제 작업**을 하고 봇 클래스는 메시지 중계 래퍼다.

```
useRoundEngine → useChat.sendMessage(prompt) → bot.sendMessage({prompt,signal})   [앱 컨텍스트]
  → DomAdapterBot.doSendMessage()                 (AbstractBot; doSendMessage가 onEvent로 스트림 제어)
    → chrome.runtime.connect({name:`dom:${llm}`}) → SEND_PROMPT(llm, prompt)   // [M-2] selector 미전송
      → background/index.ts (라우터): 대상 LLM 탭 확보(없으면 open)
        → chrome.tabs.connect(tabId) → SEND_PROMPT
          → content-script/dom-adapter.ts: (adapter-config.json[llm]으로 selector 조회)
             1) prompt textarea/contenteditable 채움 + Enter/send 클릭
             2) 응답 영역 DOM 관찰(MutationObserver) — done 판정 [G14]
             3) RESPONSE_CHUNK 누적 스트리밍 ↩ / 완료 시 DONE
          ← background ← app: onEvent(UPDATE_ANSWER) / onEvent(DONE)
```
- STOP: `signal.abort()` → 어댑터가 `{action:'STOP'}` 전송 → content script가 stop 버튼 click(또는 debounce 종료).
- 미로그인/탭없음: content script가 `{action:'ERROR', code:'ERR_002'/...}` → `ChatError(.., MISSING_HOST_PERMISSION 등)` → 셀에 "로그인 필요".

> **함의**: 6종 전부 background+content-script+탭 plumbing(메시지 라우팅)을 동일하게 사용한다 — 217에서 `DomAdapterBot` 베이스로 공통화. 내부 API/proxyFetch 경로는 존재하지 않는다.

## 4. 메시지 인터페이스 적용

원안 메시지 인터페이스는 Popup↔Background↔ContentScript 액션을 정의하나, **엔진이 앱 컨텍스트**이므로 대부분의 액션(START_SESSION/SEND_PROMPT/NEXT_ROUND/STOP_ROUND/SWAP_LLM/SAVE_MD/secret)은 **앱 내부 함수 호출/atom 갱신**로 단순화된다(`use-round-engine.ts` 메서드). chrome 메시지는 **DOM 봇 ↔ background ↔ content script 구간에만** 사용한다.

| 액션 | 구현 위치 |
|--------------|-----------|
| START_SESSION / SEND_PROMPT / NEXT_ROUND / STOP_ROUND / RESUME / SWAP_LLM | `useRoundEngine` 메서드(앱) |
| SAVE_MD | `md-export.ts`(앱) + 저장 다이얼로그 |
| START/SEND SECRET_CHAT | `secretChatAtom` + 비밀채팅 어댑터 호출(앱) |
| (DOM) SEND_PROMPT/STOP/EXTRACT_RESPONSE/RESPONSE_CHUNK | `dom-adapter.ts` ↔ background ↔ DomAdapterBot |

에러코드 ERR_001~005 → `src/utils/errors.ts`의 `ErrorCode`에 매핑 추가(217 §6).

## 5. 제어 흐름 (1 라운드)

```
START(maxRounds, tokenLimit, slots) → status=ROUND_ACTIVE
사용자 prompt 입력 → SEND_PROMPT
  status=WAITING
  for each slot.bot (병렬):
     context = buildContextForLlm(slot.botId, prevRound, roundNo, userPrompt, participants)   // 216; 1라운드는 타 LLM 키만 빈 값 [M-3]
     fullPrompt = compose(userPrompt, context, tokenLimit)  // 216
     slot.chat.sendMessage(fullPrompt)                      // [M1] useChat 래퍼(string 인자); 내부 AbortController로 스트리밍 → 셀 갱신 (engine은 bot.sendMessage 직접 호출 안 함)
  모든 슬롯 resolve → 응답을 store에서 수집(215 §4) → status=ROUND_COMPLETE
     history.push({round, userPrompt, responses})           // [M2] STOP 전 자연완료만 completed, 부분/중단 응답은 폐기
     storage.session 스냅샷 저장
  currentRound < maxRounds → status=ROUND_COMPLETE → (사용자 NEXT_ROUND = sendPrompt 재호출) status=WAITING   // [H-1] ROUND_ACTIVE 경유 생략
  currentRound == maxRounds → status=DONE → 저장 다이얼로그
STOP(언제든) → slot.chat.stopGenerating() 전체(abort) → status=STOPPED (완료분 유지, 미완료/부분 폐기)
  RESUME → ROUND_ACTIVE / END → DONE 또는 IDLE
```
